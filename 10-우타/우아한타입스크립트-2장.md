## 1. 타입이란 — 집합으로서의 타입

타입은 **값의 집합**이다. `number` 타입의 값은 "모든 숫자의 집합"이고, `"apple" | "banana"` 타입의 값은 정확히 두 개 원소를 가진 집합.

### 왜 이 관점이 중요한가

```ts
type A = number;
type B = 42;
// B ⊂ A (42는 number의 부분집합)
```

- **"좁은 타입"** = 원소가 적은 집합 (리터럴, 유니온 중 일부)
- **"넓은 타입"** = 원소가 많은 집합 (`number`, `string`, `any`)
- **할당 가능(assignable)** = "오른쪽이 왼쪽의 부분집합인가?"

Type challenges 풀면서 느꼈을 텐데, TS의 모든 타입 연산은 **집합 연산**으로 환원됨.

| 연산 | 집합 의미 |
| --- | --- |
| `A \| B` (유니온) | A ∪ B (합집합) |
| `A & B` (인터섹션) | A ∩ B (교집합) |
| `A extends B` | A ⊆ B인가? |
| `never` | ∅ (공집합) |
| `unknown` | 전체집합(이자 최상위) |

## 2. 정적 타이핑 vs 동적 타이핑, 강타입 vs 약타입

이스타가 흔히 섞어서 생각할 수 있는 두 축.

### 축 1: 타입 결정 시점

- **정적(static)**: 컴파일 타임에 타입 결정. TS, Java, C++
- **동적(dynamic)**: 런타임에 타입 결정. JS, Python, Ruby

### 축 2: 암묵적 타입 변환 허용 여부

- **강타입(strong)**: 서로 다른 타입 연산 시 에러. TS, Python, Ruby
- **약타입(weak)**: 내부적으로 타입을 바꿔 연산. JS, C++, Java

### 빼기(-) 연산으로 분류

```ts
// JS (약타입, 동적)
"2" - 1   // 1 ← 문자열을 숫자로 변환함

// Python (강타입, 동적)
"2" - 1   // TypeError

// TS (강타입, 정적)
"2" - 1   // 컴파일 타임 에러
```

**JS가 약타입이라는 게 TS의 존재 이유**. `3 + []` 같은 무의미한 연산이 런타임에 에러 없이 통과하는 걸 막으려면, 컴파일 타임에 타입을 검사해야 함.

## 3. 점진적 타이핑 — TS의 존재 이유

### 점진적(gradual) 타이핑이란

프로그램의 **일부 부분만 정적 타입 검사**를 받고, 나머지는 **동적 타입 검사**로 넘기는 시스템.

```ts
let a; // 타입 명시 X → any로 추론 (동적)
let b: number = 3; // 타입 명시 O → 정적

a = "hello"; // ✅ 허용 (any라서)
a = 42;      // ✅ 허용
b = "hello"; // 🚨 에러 (정적)
```

### 왜 점진적인가

JS → TS 마이그레이션은 코드 전체를 **한 번에** 타입 지정하는 게 현실적으로 불가능. 10만 줄짜리 코드베이스를 한 번에 다 타입 붙이는 건 비용이 너무 큼. **"지금 이 파일부터", "지금 이 함수부터"** 점진적으로 타입을 추가할 수 있어야 함.

→ 그래서 TS는 `any`를 허용함. "이 부분은 아직 타입 안 붙였어"를 표현할 수 있어야 하니까.

### 실무 팁

- **신규 프로젝트**: `strict: true`, `noImplicitAny: true`로 처음부터 엄격하게
- **기존 프로젝트 마이그레이션**: 일단 any 허용하면서 파일 단위로 점진 전환
- **`any`는 목표가 아니라 임시 방편**

## 4. 구조적 서브타이핑 — 🔴 핵심

TS의 가장 핵심 철학. 이스타가 type challenges 풀면서도 **"왜 구조만 같으면 호환되지?"** 싶었던 부분.

### 명목적(Nominal) 타이핑 — Java/C++

**타입의 이름**으로 구분. 구조가 같아도 이름이 다르면 다른 타입.

```java
class Cat { String name; }
class Arrow { String name; }

Cat cat = new Arrow(); // 🚨 에러: 이름이 달라
```

### 구조적(Structural) 타이핑 — TS

**타입의 구조(멤버 목록)**로 구분. 이름이 달라도 구조가 같으면 호환.

```ts
interface Cat {
  name: string;
}

interface Arrow {
  name: string;
}

const arrow: Arrow = { name: "화살" };
const cat: Cat = arrow; // ✅ 통과! 구조가 같음

// 함수 매개변수도 구조로 판단
function hitCat(cat: Cat) { ... }
hitCat({ name: "얼룩이", extra: true }); // ✅ 통과 (extra는 무시됨)
```

### 왜 이렇게 설계됐나

TS는 **JavaScript의 슈퍼셋**. JS는 **덕 타이핑 언어**. JS 코드 스타일을 그대로 수용하려면 구조로 판단하는 게 자연스러움.

> "이게 오리처럼 걷고 오리처럼 꽥꽥대면, 그건 오리다."

### React에서 구조적 서브타이핑이 의미하는 것

```tsx
// Props 정의
interface ButtonProps {
  onClick: () => void;
  label: string;
}

// 사용
const props = {
  onClick: () => alert("hi"),
  label: "click me",
  extraStuff: 42, // 필요 없는 속성
};

// ✅ 통과: ButtonProps의 구조를 "포함"하면 됨
<Button {...props} />
```

단, **객체 리터럴을 직접 전달**할 때는 "초과 속성 검사(excess property check)"가 작동함.

```tsx
<Button onClick={...} label="..." extraStuff={42} />
// 🚨 에러: extraStuff는 ButtonProps에 없음
```

→ 변수로 한 단계 거치면 통과, 리터럴로 직접 넣으면 에러. 이 차이가 실무에서 간혹 혼란의 원인.

## 5. 덕 타이핑 vs 구조적 타이핑 — 닮았지만 다름

둘 다 "구조로 타입 판단"이라 헷갈리기 쉬움.

| | 덕 타이핑 (JS, Python) | 구조적 타이핑 (TS) |
| --- | --- | --- |
| 검사 시점 | **런타임** | **컴파일 타임** |
| 동적/정적 | 동적 | 정적 |
| 표현 방식 | 암묵적 | 명시적 (interface, type) |

### 핵심 차이

JS는 코드 실행 중에 "이 객체가 `name` 속성이 있네? 그럼 사용 가능"이라고 판단. 런타임에 없으면 바로 crash.

TS는 코드 작성 시점에 "이 변수의 타입은 이런 구조를 가져야 해"를 확인. 런타임에는 이미 타입 정보가 사라졌음(JS로 컴파일됨).

## 6. 값 공간 vs 타입 공간

TS에서 가장 처음 헷갈리는 지점.

### 같은 이름이 값/타입 양쪽에 살 수 있음

```ts
class User {
  constructor(public name: string) {}
}

const u1: User = new User("kim"); // User = 타입
const u2 = new User("lee");       // User = 값 (생성자 함수)
```

`class`, `enum`은 선언 시 **값 공간과 타입 공간 양쪽에 이름을 등록**함.

### `type`, `interface`는 타입 공간에만

```ts
type UserType = { name: string };
const u3 = new UserType(); // 🚨 에러: UserType은 런타임에 존재 안 함
```

### `typeof` 연산자의 두 얼굴

```ts
// JS typeof (런타임, 값 공간)
typeof "hello"  // "string"

// TS typeof (타입 공간)
const obj = { a: 1, b: 2 };
type ObjType = typeof obj;  // { a: number; b: number }
```

**이스타가 쓴 `typeof arr[number]`**가 바로 **타입 공간의 typeof + 타입 공간의 인덱스드 액세스**. 런타임 JS 문법과 똑같이 생겼지만 완전히 다른 차원에서 동작.

### 실무 팁: `as const` + `typeof`

```ts
const CATEGORIES = ["한식", "중식", "일식"] as const;
type Category = (typeof CATEGORIES)[number];
// "한식" | "중식" | "일식"
```

- `CATEGORIES`는 **값** (런타임에 존재하는 배열)
- `typeof CATEGORIES`는 **타입** (readonly ["한식", "중식", "일식"])
- `[number]`는 **타입 공간에서 "모든 숫자 인덱스로 꺼낼 수 있는 원소"**

값과 타입을 동시에 쓸 수 있어서 SSOT(Single Source of Truth) 구현에 유용.

## 7. 타입 애너테이션 — 두 가지 방식

### A. 변수에 직접

```ts
const name: string = "kim";
const count: number = 0;
```

### B. 함수 시그니처에

```ts
function greet(name: string): string {
  return `hello, ${name}`;
}

const greet2 = (name: string): string => `hello, ${name}`;
```

### 실무 권장: "명시하지 마라, 추론을 믿어라"

```ts
// ❌ 과한 애너테이션
const name: string = "kim";  // 추론해도 string

// ✅ 추론에 맡기기
const name = "kim";  // "kim" 리터럴 타입으로 추론

// ✅ 추론이 의도와 다를 때만 명시
let status: "pending" | "done" = "pending";
//     ↑ let이라 "pending"이 아니라 string으로 추론되는 걸 막음

// ✅ 함수 반환 타입은 명시 권장 (API 계약)
function fetchUser(id: string): Promise<User> { ... }
```

## 🎯 이스타에게 필요한 액션 요약

| 상황 | 이렇게 해 |
| --- | --- |
| JS 파일을 TS로 옮길 때 | 한 번에 다 바꾸지 말고 파일 단위 점진 전환 |
| `any` 만나면 | "임시 방편이다. 언젠가 지운다"고 생각 |
| 구조가 같은데 타입 에러 나면 | 객체 리터럴 직접 전달인지 확인 (excess property check) |
| `typeof` 쓸 때 | 값 공간인지 타입 공간인지 구분 |
| 변수 타입 명시 | 추론에 맡기되, **함수 반환 타입**은 명시 |
| enum/class | 값 + 타입 양쪽에 등록된다는 점 기억 |
