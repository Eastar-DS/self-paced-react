## 1. any, unknown — "아무거나"의 두 얼굴

### any — 타입 검사 포기 선언

```ts
let x: any = 3;
x = "hello";
x = { a: 1 };
x.foo.bar.baz;  // ✅ 에러 안 남. 런타임에서 터짐
```

- 할당·사용 둘 다 자유
- **타입 시스템을 끄는 스위치**
- 전염성: `any` 값을 받으면 그 값도 any 됨

### unknown — 타입을 모르는 값 (안전한 any)

```ts
let x: unknown = 3;
x = "hello";    // ✅ 할당은 자유
x.foo;          // 🚨 에러: 사용 전에 타입을 좁혀야 함

if (typeof x === "string") {
  x.toUpperCase();  // ✅ 좁혀진 뒤라 안전
}
```

- 할당은 any처럼 자유
- **사용은 타입을 좁힌 뒤에만** → 안전성 유지

### 언제 뭘 쓰나

| 상황 | 선택 |
| --- | --- |
| 외부 API 응답, 사용자 입력 | `unknown` (검증 후 사용) |
| 타입 지정이 아직 어려운 임시 단계 | `any` (빨리 지울 것) |
| 리덕스 액션 페이로드, 이벤트 데이터 | `unknown` (타입 가드로 좁혀서) |

### React에서

```ts
// ❌ 나쁨
const handleError = (err: any) => console.log(err.message);
// err이 null일 수도 있는데 검증 없이 .message 접근

// ✅ 좋음
const handleError = (err: unknown) => {
  if (err instanceof Error) {
    console.log(err.message);
  }
};
```

## 2. void, never — "없음"의 두 종류

### void — 반환값이 없음 (undefined 자동 반환)

```ts
function logHello(): void {
  console.log("hello");
}
```

**실무 함정**: callback 시그니처의 void

```ts
type Handler = (value: string) => void;

// void 반환이라고 했는데 뭔가 반환하는 함수도 받아줌
const h: Handler = (v) => 42;  // ✅ 통과

// void prop에 어떤 반환값이든 넣을 수 있는 이유:
// "반환값을 사용하지 않겠다"는 의미지, "반환하지 마라"가 아님
```

### never — 절대 도달할 수 없음 (공집합)

```ts
// 항상 에러 던지는 함수
function fail(): never {
  throw new Error();
}

// 무한 루프
function spin(): never {
  while (true) {}
}
```

### never의 실전 활용: **완전성 검사**

```ts
type Shape = Circle | Square | Triangle;

function area(s: Shape): number {
  switch (s.kind) {
    case "circle":   return Math.PI * s.r ** 2;
    case "square":   return s.side ** 2;
    case "triangle": return (s.base * s.height) / 2;
    default:
      const exhaustive: never = s;
      // 🚨 Shape에 새 종류 추가 시 여기서 에러 → 까먹는 일 방지
      return exhaustive;
  }
}
```

이거 type challenges에서는 잘 안 나오지만 **실무에서 굉장히 유용**. discriminated union 쓸 때 반드시 기억.

## 3. Array vs Tuple

### Array — 길이 고정 안 됨, 모든 원소 같은 타입

```ts
const nums: number[] = [1, 2, 3];
const strs: Array<string> = ["a", "b"];
```

### Tuple — 길이 고정, 위치별 타입 다름

```ts
const pair: [string, number] = ["age", 30];

// React useState가 튜플을 반환하는 대표 사례
const [count, setCount] = useState(0);
// → [number, React.Dispatch<...>]
```

### 읽기 전용 (as const)

```ts
const arr = [1, 2, 3] as const;  // readonly [1, 2, 3]
```

- `as const` 쓰면 **튜플로 고정 + readonly + 리터럴 타입**
- 이스타가 이미 SSOT 패턴에서 쓰고 있는 바로 그것

## 4. Enum — 🔥 배민이 권장하지 않는 이유

### enum이란

```ts
enum Items {
  DRINK = 1,
  MAIN = 2,
}

Items.DRINK;       // 1
Items[1];          // "DRINK" (역참조!)
```

### 문제점 1: 역참조 허용 → 타입 안전성 구멍

```ts
const x: Items = 99;  // 🚨 에러는 나야 할 것 같은데 안 나는 경우도 있음
// numeric enum은 어떤 숫자든 통과
```

### 문제점 2: **Tree-shaking 안 됨**

enum은 컴파일 시 **IIFE(즉시 실행 함수)**로 변환됨:

```js
// 컴파일 결과
(function (Items) {
  Items[Items["DRINK"] = 1] = "DRINK";
  Items[Items["MAIN"] = 2] = "MAIN";
})(Items || (Items = {}));
```

번들러(Rollup 등)는 IIFE를 **"side effect 있는 코드"** 로 판단해서 사용 안 해도 못 없앰. 결과적으로 **번들 크기 증가**.

### 문제점 3: TS 고유 기능 → JS와 비대칭

자바스크립트에 없는 개념이라, TS 지우면 문법이 동작 안 함. **Node.js가 타입 어노테이션만 stripping**하는 트렌드와 역행.

### 대안: `as const` 객체

```ts
// ❌ enum
enum Category {
  KOREAN = "한식",
  CHINESE = "중식",
}

// ✅ as const
const CATEGORY = {
  KOREAN: "한식",
  CHINESE: "중식",
} as const;

type Category = (typeof CATEGORY)[keyof typeof CATEGORY];
// "한식" | "중식"
```

- tree-shakable (일반 객체라서)
- JS에 그대로 존재 (TS 지워도 동작)
- 타입 추론 정확

**배민/토스 등 대부분 FE 회사가 `enum` 지양 권장**. 우테코 레벨2 과정에서도 리뷰어가 enum보다 `as const` 추천하는 경우 많음.

## 5. object, `{}`, Object — 셋의 차이

헷갈리기 쉬운 3종. 실무에서 대부분 **셋 다 안 쓰는 게 좋음**.

```ts
let a: object;   // 원시 타입을 제외한 모든 것
let b: {};       // null, undefined 빼고 다 됨 (사실상 any 비슷)
let c: Object;   // {}와 거의 동일, JavaScript의 Object 래퍼

a = { x: 1 };    // ✅
a = 1;           // 🚨 에러 (원시값 아님)

b = { x: 1 };    // ✅
b = "hello";     // ✅ (primitive도 통과!)
b = null;        // 🚨 에러
```

### 실무 권장

- **"뭐든 객체"** → 구체적 인터페이스 정의 (`{ [key: string]: unknown }`)
- 절대 `{}`나 `Object` 쓰지 말 것 (린트가 잡아줌: `@typescript-eslint/ban-types`)

## 6. 유니온 `|` vs 교차 `&`

Type challenges에서 많이 다뤘을 주제. 한 번 정리.

### 유니온 (`|`) — OR, 합집합

```ts
type StringOrNumber = string | number;
let x: StringOrNumber = "hello";
x = 42;  // ✅ 둘 다 됨
```

### 교차 (`&`) — AND, 교집합

```ts
type A = { a: string };
type B = { b: number };
type AB = A & B;  // { a: string; b: number }
```

### 함정: 유니온 객체의 속성 접근

```ts
type Dog = { bark: () => void; name: string };
type Cat = { meow: () => void; name: string };
type Pet = Dog | Cat;

const pet: Pet = ...;
pet.name;    // ✅ 둘 다 공통
pet.bark();  // 🚨 에러: Cat에는 bark 없음

// 좁혀야 사용 가능
if ("bark" in pet) {
  pet.bark();  // ✅
}
```

### Discriminated Union (태그된 유니온)

```ts
type Shape =
  | { kind: "circle"; radius: number }
  | { kind: "square"; side: number };

function area(s: Shape): number {
  switch (s.kind) {
    case "circle": return Math.PI * s.radius ** 2;
    case "square": return s.side ** 2;
  }
}
```

React에서 상태 관리(로딩/성공/에러)할 때 자주 쓰는 패턴.

## 7. type vs interface — 실전 선택 기준

겉모습 비슷하지만 세부가 다름.

| | `type` | `interface` |
| --- | --- | --- |
| 원시 타입 별칭 | ✅ (`type ID = string`) | ❌ |
| 유니온/교차 | ✅ | ❌ (extends로 교차만) |
| 확장 | `&`로 교차 | `extends` |
| **선언 병합** | ❌ | ✅ |
| 튜플/함수 타입 | 자연스러움 | 가능하지만 어색 |

### 선언 병합(Declaration Merging)

```ts
interface User { name: string; }
interface User { age: number; }

const u: User = { name: "kim", age: 30 };  // ✅ 병합됨

// type은 같은 이름 선언 시 에러
type User2 = { name: string };
type User2 = { age: number };  // 🚨 에러
```

### 실전 선택 가이드 (배민 컨벤션 참고)

- **외부 라이브러리 타입 확장** (예: styled-components의 DefaultTheme) → `interface` (선언 병합 필요)
- **유니온/교차/유틸리티 조합** → `type`
- **객체 형태 Props 정의** → 취향. 팀 컨벤션 따르기
- **둘 다 가능하면** → `type` (더 유연)

우아한형제들 컨벤션은 **`type` 선호** (일관성과 유연성).

## 8. 인덱스 시그니처

"속성 이름은 모르지만 값 타입은 안다" 할 때 사용.

```ts
interface PhoneBook {
  [name: string]: string;
}

const book: PhoneBook = {
  "김수": "010-1111-2222",
  "이영": "010-3333-4444",
};
```

### 다른 속성과 같이 쓸 때 주의

```ts
interface Mixed {
  [key: string]: number | boolean;
  length: number;   // ✅ number
  isValid: boolean; // ✅ boolean
  name: string;     // 🚨 에러: string은 인덱스 시그니처 타입에 포함 안 됨
}
```

인덱스 시그니처에서 **명시 속성들은 인덱스의 값 타입에 포함되어야 함**.

### 실무에서 왜 자주 안 쓰나

```ts
const book: PhoneBook = { "김수": "010-..." };
book["없는이름"].toUpperCase();  
// 🚨 런타임 에러. 타입은 string인데 undefined가 반환됨
```

**`noUncheckedIndexedAccess`** 옵션을 켜면 인덱스 결과가 `string | undefined`로 나와 안전해짐. `Record<K, V>`를 쓰면 K가 유한 집합이라 더 안전.

## 9. 인덱스드 액세스 타입 — 🔴

**이스타가 `typeof arr[number]`에서 `number`가 뭔지 궁금해했던 그것.**

### 기본 형태

```ts
type Example = {
  a: number;
  b: string;
  c: boolean;
};

type A = Example["a"];           // number
type AB = Example["a" | "b"];    // number | string
type All = Example[keyof Example]; // number | string | boolean
```

**키 자리에 타입이 들어감.** JS의 `obj["a"]`와 형태는 같지만 **타입 공간에서 동작**.

### 배열에서 `[number]`의 정체

```ts
const fruits = ["apple", "banana", "cherry"] as const;
// 타입: readonly ["apple", "banana", "cherry"]

type FruitArr = typeof fruits;
// → readonly ["apple", "banana", "cherry"]

type Fruit = FruitArr[number];
// → "apple" | "banana" | "cherry"
```

**왜 `[number]`?**

배열의 **모든 인덱스는 number 타입**. 그래서 `[number]`는 "인덱스가 number인 모든 자리의 타입" = **모든 원소의 타입 유니온**.

- `fruits[0]` → `"apple"`
- `fruits[1]` → `"banana"`
- `fruits[2]` → `"cherry"`
- `fruits[number]` → `"apple" | "banana" | "cherry"` (모두 합친 것)

직관적으로 "**숫자 인덱스로 꺼낼 수 있는 모든 것의 유니온**"이라고 기억.

### 실전 패턴

```ts
// 배민 실무 예시
const PROMOTIONS = [
  { type: "product", name: "chicken" },
  { type: "product", name: "pizza" },
  { type: "card", name: "cheer-up" },
] as const;

type Promotion = (typeof PROMOTIONS)[number];
// { readonly type: "product"; readonly name: "chicken" }
// | { readonly type: "product"; readonly name: "pizza" }
// | { readonly type: "card"; readonly name: "cheer-up" }

type PromotionType = Promotion["type"];  // "product" | "card"
```

## 10. Mapped Types

Type challenges 3번(`Readonly`)이 mapped type의 시작.

### 기본 문법

```ts
type Readonly<T> = {
  readonly [K in keyof T]: T[K];
};

type Partial<T> = {
  [K in keyof T]?: T[K];
};
```

### 수정자 추가/제거

```ts
type Mutable<T> = {
  -readonly [K in keyof T]: T[K];  // readonly 제거
};

type Required<T> = {
  [K in keyof T]-?: T[K];  // optional 제거
};
```

### as 재매핑 (TS 4.1+)

```ts
type Getters<T> = {
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K];
};

interface User {
  name: string;
  age: number;
}

type UserGetters = Getters<User>;
// { getName: () => string; getAge: () => number }
```

이스타가 10번까지 풀었으면 여기까지는 경험이 있을 것. **실무에서는 유틸리티 타입(Pick, Omit, Partial, Required)을 쓰는 것만으로 충분한 경우가 많음.**

## 11. Template Literal Types

문자열 리터럴을 조합하는 타입. TS 4.1+.

```ts
type Direction = "top" | "bottom" | "left" | "right";
type Position = `${Direction}-${number}`;
// "top-0" | "top-1" | ... (실제로는 number가 무한이라 리터럴 아님)

type CssVar = `--${string}`;
const color: CssVar = "--primary-color";  // ✅
const wrong: CssVar = "primary-color";    // 🚨 에러
```

### 실무 활용

```ts
// 이벤트 이름 타입 생성
type Event = "click" | "hover";
type Handler = `on${Capitalize<Event>}`;
// "onClick" | "onHover"

// API 엔드포인트 타입
type ApiVersion = "v1" | "v2";
type Resource = "users" | "posts";
type Endpoint = `/api/${ApiVersion}/${Resource}`;
// "/api/v1/users" | "/api/v1/posts" | ...
```

## 12. 제네릭 — `.tsx` 함정까지

### 기본

```ts
function identity<T>(arg: T): T {
  return arg;
}

identity<string>("hello");  // 명시
identity(42);                // 추론 (number)
```

### 제약(extends) — 🔴 `.tsx`에서 필수

```tsx
// ❌ .tsx에서 에러: <T>가 JSX 태그로 파싱됨
const arrow = <T>(arg: T): T[] => [...];

// ✅ 방법 1: extends
const arrow = <T extends {}>(arg: T): T[] => [...];
const arrow = <T extends unknown>(arg: T): T[] => [...];

// ✅ 방법 2: 쉼표 트릭 (호불호 갈림)
const arrow = <T,>(arg: T): T[] => [...];

// ✅ 방법 3: function 선언문
function identity<T>(arg: T): T[] { return [...]; }
```

**이스타의 useForm에서 `<T extends Record<string, string>>` 쓰고 있는 이유**: JSX 충돌 + 의미 있는 제약 둘 다 해결.

### 기본값

```ts
interface Props<T = HTMLElement> {
  target: T;
}

// 안 넘기면 HTMLElement 사용
type DefaultProps = Props;
```

### 여러 타입 변수

```ts
function pair<A, B>(a: A, b: B): [A, B] {
  return [a, b];
}

const p = pair("hello", 42);  // [string, number]
```

### Conditional Type + infer (type challenges 고급)

```ts
type ReturnType<T> = T extends (...args: any[]) => infer R ? R : never;

type R = ReturnType<() => string>;  // string
```

`infer`는 **"이 자리에 오는 타입을 R이라는 이름으로 잡아라"**. type challenges 20번대 가면 infer 떡잡이로 등장.

## 🎯 이스타에게 필요한 액션 요약

| 상황 | 이렇게 해 |
| --- | --- |
| 외부 응답 타입 모름 | `unknown` + 타입 가드 (any 쓰지 마) |
| discriminated union 사용 | `default: const _: never = x` 로 완전성 검사 |
| Enum 쓰고 싶다 | **`as const` 객체로 대체** |
| `object`, `{}`, `Object` | **모두 피하고** 구체 타입 정의 |
| 팀 컨벤션 없다면 | `type` 선호 (유연) |
| 인덱스 시그니처 | `noUncheckedIndexedAccess` 옵션 켜기 or `Record<K, V>` |
| 배열 원소 타입 추출 | `(typeof arr)[number]` 관용구 |
| `.tsx`에서 제네릭 함수 | `<T extends ...>` 형태 필수 |

---

## 🔥 우선순위 복습 (이스타 기준)

1. **Enum 대체 패턴 (`as const` + keyof + typeof)** — 우테코 레벨2 리뷰에서 자주 지적됨
2. **인덱스드 액세스 `[number]`의 의미** — 이스타가 직접 질문한 것 🔴
3. **never로 완전성 검사** — discriminated union에서 필수
4. **any vs unknown의 사용 지침** — 실무 마인드
5. **`.tsx`에서 제네릭 함수 문법** — useForm에서 이미 쓰고 있음
6. **type vs interface 선택 기준** — 팀 컨벤션 맞추기
