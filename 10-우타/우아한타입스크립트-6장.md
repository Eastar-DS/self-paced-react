## 1. 런타임 vs 컴파일타임

| 시점 | 무엇이 일어나는가 |
| --- | --- |
| **컴파일타임** | 소스코드 → 컴퓨터가 이해할 수 있는 기계어/바이트코드로 변환되는 과정 |
| **런타임** | 컴파일된 프로그램이 메모리에 올라가 실행되는 시점 |

**JS는 인터프리터 언어로 알려져 있지만** V8 엔진 같은 현대 엔진은 최적화를 위해 내부적으로 컴파일 단계(JIT)를 거침. 다만 개발자가 직접 컴파일을 실행하지 않을 뿐.

---

## 2. 자바스크립트 런타임 구성

JS 런타임 = JS가 실행되는 환경 (크롬, Node.js 등).

주요 구성:
- **JS 엔진** (V8, SpiderMonkey 등) — 코드 파싱·실행
- **Web API** — DOM, setTimeout, fetch (브라우저) / fs, http (Node.js)
- **콜백 큐** — 비동기 콜백 대기 장소
- **이벤트 루프** — 콜 스택 비면 콜백 큐에서 꺼내 실행
- **렌더 큐** (브라우저) — 렌더링 작업 대기

런타임 에러 예시 — **이런 건 TS가 막을 수 없음**:
```js
let foo;
foo.bar;  // 🚨 TypeError (런타임에서 발생)

const arr = null;
arr.length;  // 🚨 TypeError
```

> **포인트:** TS는 **컴파일타임 정적 분석기**. 런타임 `undefined`/`null` 접근을 타입으로 잡아주긴 하지만, 타입을 거짓말하면(`as`, `!`, `any`) 런타임 에러가 그대로 뚫고 나옴.

## 3. TS 컴파일 ≒ 트랜스파일

일반적인 컴파일: 고수준 언어 → 저수준 언어 (C → 어셈블리).

**TS 컴파일은 고수준 → 고수준 (TS → JS).** 엄밀히는 **트랜스파일**이라 부르는 게 정확. TS 컴파일러(tsc)는 **source-to-source compiler**.

비슷한 트랜스파일러:
- Babel: ES6+ → ES5
- Emscripten: C/C++ → JS

하지만 관습적으로 "TS 컴파일"이라고 통칭.

---

## 4. 🔴 6단계 컴파일 파이프라인

TS 소스코드가 실행되기까지 거치는 **6단계**:

```
[작성 시점]                          [런타임]
 ↓                                   ↓
1. TS 소스 → TS AST   (tsc)
2. TS AST → 타입 검사   (tsc)  ← 타입 정보는 여기까지만
3. TS AST → JS 소스   (tsc)
                              ┌─────────────
                              │ 4. JS 소스 → JS AST   (V8 등)
                              │ 5. JS AST → 바이트코드
                              │ 6. 바이트코드 실행
                              └─────────────
```

**1~3단계:** tsc가 담당. 타입 정보가 존재.
**4~6단계:** 런타임 엔진(V8/Node)이 담당. **타입 정보가 완전히 사라진 JS만 받음.**

---

## 5. 🔴 "타입은 3단계부터 사라진다"의 의미

### 핵심 사실

> TS 타입은 1~2단계에서만 존재. 3단계(TS→JS 변환)부터는 타입이 주석처럼 지워짐.

### 결과: 런타임에는 타입이 없음

```ts
interface User { name: string; age: number }

function isUser(x: unknown): x is User {
  // 🚨 이렇게 쓸 수 없음:
  // return x instanceof User;  // ERROR: User는 값이 아님 (타입은 런타임에 없음)

  // ✅ 직접 구조를 검사해야 함:
  return typeof x === "object" && x !== null
    && typeof (x as any).name === "string"
    && typeof (x as any).age === "number";
}
```

### 그래서 필요한 것들
- **Superstruct / Zod / io-ts** — 런타임 스키마 검증 라이브러리
- **사용자 정의 타입 가드 (`is`)** — 4장에서 본 그것
- **Discriminated Union** — 런타임에 실제로 존재하는 필드로 분기

### 자바스크립트에 남는 것 / 사라지는 것

| 남음 | 사라짐 |
| --- | --- |
| `class`, `enum` (값으로도 쓰임) | `interface`, `type` |
| `const`, `let`, 함수 본문 | 타입 annotation (`: string`) |
| `Object.keys`, `typeof` | `keyof`, TS `typeof`(타입 연산자) |

> **enum은 JS 런타임에 IIFE + 객체로 남아 있음** (3장에서 본 tree-shaking 문제의 이유).

## 6. AST — 소스코드를 트리로

**Abstract Syntax Tree**: 컴파일러가 소스코드를 분석하면서 만드는 **트리 구조 데이터**.

```js
const x = 1 + 2;
```

```
VariableDeclaration
└── VariableDeclarator
    ├── Identifier: "x"
    └── BinaryExpression: "+"
        ├── NumericLiteral: 1
        └── NumericLiteral: 2
```

**어휘적 분석(lexical analysis)** → 토큰화 → **구문 분석(syntax analysis)** → AST.

AST를 알아야 하는 실전 맥락:
- ESLint 커스텀 룰 작성
- Babel 플러그인
- IDE 자동 완성/리팩터링
- TS 컴파일러의 2단계(타입 검사)가 AST를 순회하면서 수행

## 7. tsc 동작 4단계

tsc 내부는 크게 4개 컴포넌트로 구성:

1. **Scanner (스캐너)** — 소스코드 → 토큰
2. **Parser (파서)** — 토큰 → AST
3. **Binder (바인더)** — AST의 각 선언을 심볼(Symbol)과 연결
4. **Checker (체커)** — 심볼 기반으로 타입 검사
5. **Emitter (에미터)** — AST → JS 코드 출력

심볼(Symbol)이란?
- 식별자(변수/함수/타입 이름)의 **선언 정보 단위**
- 한 파일 안의 `const x`와 다른 파일의 `x`는 서로 다른 심볼
- 같은 이름이라도 스코프가 다르면 다른 심볼

> **이스타가 실무에서 만날 곳:** 에디터에서 "정의로 이동(F12)"이 작동하는 원리가 이 바인더/체커의 심볼 추적.

## 8. tsconfig 주요 옵션 (레벨2 관점)

레벨2에서 실제로 건드리는 옵션만.

### 필수 / 자주 보는 것

```jsonc
{
  "compilerOptions": {
    "target": "ES2020",           // 변환 후 JS 버전
    "module": "ESNext",           // 모듈 시스템 (import/export)
    "moduleResolution": "bundler",// vite/webpack 쓸 때
    "jsx": "react-jsx",           // React 17+ 자동 JSX런타임
    "strict": true,               // ★ 절대 끄지 마세요
    "noEmit": true,               // 변환은 Vite가 함. tsc는 타입체크만
    "isolatedModules": true,      // 파일 단독 트랜스파일 가능 보장 (Vite 요구)
    "skipLibCheck": true,         // node_modules 타입체크 skip (속도)
    "resolveJsonModule": true,    // JSON import 허용
    "lib": ["ES2020", "DOM", "DOM.Iterable"]
  },
  "include": ["src"]
}
```

### `strict: true`에 포함된 핵심 하위 옵션
- `noImplicitAny` — 타입 없는 곳에 `any` 자동 부여 금지
- `strictNullChecks` — `null`/`undefined` 엄격 구분
- `strictFunctionTypes` — 함수 파라미터 반공변 체크
- `strictBindCallApply` — `bind/call/apply` 인자 체크
- `noImplicitThis` — `this` 타입 명시 강제
- `alwaysStrict` — `"use strict"` 자동

### 레벨2에서 켜두면 좋은 추가 옵션
- `noUncheckedIndexedAccess` — `arr[0]`이 `T | undefined`로 좁혀짐 (런타임 안전)
- `noImplicitOverride` — 상속 메서드 override 명시 강제
- `exactOptionalPropertyTypes` — `foo?: string`과 `foo?: string | undefined` 구분

## 📎 레벨2에서 바로 써먹을 것

| 주제 | 언제 / 왜 |
| --- | --- |
| **"타입은 런타임에 없다"** | API 응답 검증, 폼 입력 검증은 **코드를 따로 짜야 함** |
| **타입 가드 함수의 존재 이유** | `interface`만으론 `instanceof`가 안 됨 → `is` 필요 |
| **strict 옵션** | `tsconfig.json` 켜면 미션에서 타입 품질 자동 상승 |
| **isolatedModules** | Vite/ESBuild 환경에서 타입-only 재export에 `export type` 쓰는 이유 |
