## 1. ReactElement vs ReactNode vs JSX.Element

**🔴 지금 Modal.tsx에서 `children: ReactNode` 쓰고 있는 그거.** 왜 이 셋 중에 ReactNode였는지 알아야 함.

### 포함 관계

```
ReactNode (가장 넓음)  ⊇  JSX.Element  ⊇  ReactElement (가장 구체적)
```

### 각 타입의 정체

#### `ReactElement` — createElement 호출의 결과 객체

```ts
interface ReactElement<P = any, T = ...> {
  type: T;
  props: P;
  key: string | null;
}
```

`<Button size={14} />` → `createElement(Button, { size: 14 })` → 위 모양의 **객체**.

#### `JSX.Element` — `ReactElement<any, any>`에 불과함

```ts
// @types/react 내부
namespace JSX {
  interface Element extends React.ReactElement<any, any> {}
}
```

props/type이 `any`라 타입 추론이 약함. 그냥 "JSX 결과물이다" 정도.

#### `ReactNode` — 렌더 가능한 모든 것의 유니온

```ts
type ReactNode =
  | ReactElement
  | string
  | number
  | boolean
  | null
  | undefined
  | ReactFragment
  | ReactPortal;
```

---

### 언제 뭘 쓰는가

| 상황 | 타입 | 이유 |
| --- | --- | --- |
| **children prop** | `ReactNode` | `<Modal>글자만</Modal>`, `<Modal>{null}</Modal>`도 허용해야 함 |
| icon prop처럼 **JSX 엘리먼트 하나만** 받을 때 | `JSX.Element` 또는 `ReactElement<IconProps>` | `icon.props.size` 접근 가능 |
| **props 타입을 좁혀서 추론 받고 싶을 때** | `ReactElement<IconProps>` | `icon.props.size`가 `number`로 정확히 추론됨 |

### 현재 Modal.tsx와 연결

```tsx
// src/components/Modal.tsx
interface ModalProps {
  onClose: () => void;
  children: ReactNode;  // ✅ 문자열/숫자/null 다 들어올 수 있으니 ReactNode
}
```

만약 `children: JSX.Element`로 했다면 `<Modal>hello</Modal>`에서 "string은 JSX.Element가 아니야" 에러가 남. ReactNode가 정답.

## 2. React.FC는 왜 기피되는가

**🔴 이스타가 "React.FC 안 쓰고 `({ onClose }: Props) => ...` 형태로 쓰는 이유" 깊게 알고 있어야 함.**

### 문제점 1: 암묵적 children (v17까지)

React 17까지는 `React.FC<Props>`가 **암묵적으로 children을 포함**했음.

```tsx
// v17까지
const NoChildren: React.FC<{ name: string }> = ({ name, children }) => {
  return <div>{name}{children}</div>; // children 안 쓰기로 했는데도 쓸 수 있음
};
```

→ children을 안 받을 컴포넌트에 `<NoChildren>실수</NoChildren>` 해도 에러 안 남. 의도치 않은 API 노출.

### 문제점 2: v18에서 children 제거 → API 깨짐

React 18부터 `React.FC`에서 children이 빠짐. 그래서 기존 `React.FC` 코드베이스는 업그레이드하면 children 쓰던 곳이 다 에러. `React.VFC`는 아예 **deprecated되어 삭제**됨.

### 문제점 3: 반환 타입이 고정되어 유연성↓

`React.FC`는 반환 타입이 `ReactElement | null`. 배열 반환이나 `Fragment` 반환 시 제약이 생길 수 있음.

### 모던 표준

```tsx
// ✅ 이스타가 쓰고 있는 방식 (권장)
interface ModalProps {
  onClose: () => void;
  children: ReactNode;
}
const Modal = ({ onClose, children }: ModalProps) => { ... };

// ❌ 레거시
const Modal: React.FC<ModalProps> = ({ onClose, children }) => { ... };
```

**결론**: 이스타의 현재 스타일이 맞음. `React.FC` 쓰지 마.

## 3. PropsWithChildren 유틸리티

`children`을 받는 컴포넌트를 매번 `children: ReactNode` 쓰기 귀찮을 때.

```ts
// React가 제공하는 유틸리티 타입
type PropsWithChildren<P = unknown> = P & {
  children?: ReactNode | undefined;
};
```

### 사용 예시

```tsx
interface CardOwnProps {
  title: string;
}

// 방법 A: 직접 쓰기 (이스타 현재 스타일)
interface CardProps extends CardOwnProps {
  children: ReactNode;
}

// 방법 B: PropsWithChildren 사용
type CardProps = PropsWithChildren<CardOwnProps>;
// → { title: string; children?: ReactNode | undefined }
```

### ⚠️ 주의: children이 **optional**로 들어옴

`PropsWithChildren`이 붙이는 children은 `?:` (optional)임. **children을 반드시 받아야 하는 컴포넌트**라면 직접 써야 함.

```tsx
// Modal은 children이 필수여야 말이 됨
interface ModalProps {
  onClose: () => void;
  children: ReactNode; // ← 필수로 명시 (PropsWithChildren 쓰면 optional)
}
```

### 언제 PropsWithChildren을 쓸까

- children이 있어도 그만, 없어도 그만인 **레이아웃/컨테이너 컴포넌트** (Card, Panel, Section 등)
- 반대로 **Modal, Drawer, Accordion 같은 "속 내용이 반드시 필요한"** 컴포넌트는 직접 `children: ReactNode`로 required 표기하는 게 의도가 명확

## 4. 이벤트 핸들러 타입 체계 정리

**🔴 이스타 `FormEvent<HTMLFormElement>`, `ChangeEvent<HTMLInputElement>` 쓰고 있는데 전체 지도 한 번 봐야 함.**

### 두 가지 표기 방법

#### A. 이벤트 객체 타입으로 매개변수 지정

```tsx
const handleSubmit = (e: FormEvent<HTMLFormElement>) => {
  e.preventDefault();
};
```

#### B. 핸들러 타입으로 함수 전체 지정

```tsx
const handleSubmit: FormEventHandler<HTMLFormElement> = (e) => {
  e.preventDefault(); // e는 자동으로 FormEvent<HTMLFormElement>
};
```

**실무 팁**: 매개변수만 지정(A)하는 게 많음. Contextual Typing이 작동하는 상황(JSX 인라인, props 전달)에서는 둘 다 생략 가능.

---

### 자주 쓰는 이벤트 타입 표

| 이벤트 | 이벤트 타입 | 핸들러 타입 | 주로 붙는 곳 |
| --- | --- | --- | --- |
| 폼 제출 | `FormEvent<HTMLFormElement>` | `FormEventHandler<HTMLFormElement>` | `<form onSubmit>` |
| input/textarea/select 변경 | `ChangeEvent<HTMLInputElement>` | `ChangeEventHandler<HTMLInputElement>` | `onChange` |
| 버튼 클릭 | `MouseEvent<HTMLButtonElement>` | `MouseEventHandler<HTMLButtonElement>` | `onClick` |
| 키보드 입력 | `KeyboardEvent<HTMLInputElement>` | `KeyboardEventHandler<HTMLInputElement>` | `onKeyDown` |
| 포커스 | `FocusEvent<HTMLInputElement>` | `FocusEventHandler<HTMLInputElement>` | `onBlur`, `onFocus` |

⚠️ **DOM 전역 타입과 충돌 주의**
`document.addEventListener("keydown", (e: KeyboardEvent) => ...)` 같은 **DOM 전역 리스너**는 React의 `KeyboardEvent`가 아니라 **`lib.dom.d.ts`의 전역 `KeyboardEvent`**를 씀.

Modal.tsx에서 `e: KeyboardEvent`를 import 없이 쓴 이유가 이것. React 이벤트는 `import { KeyboardEvent } from "react"` 해야 함.

### 현재 useForm과 연결

```tsx
// src/hooks/useForm.ts
const handleChange = (
  e: ChangeEvent<HTMLInputElement | HTMLTextAreaElement | HTMLSelectElement>
) => { ... };
```

**왜 유니온?** input/textarea/select 셋 다 지원하려니까. 모두 `target.name`, `target.value` 갖고 있어서 유니온으로 받아도 안전.

## 5. HTML 요소 props 상속 (ComponentPropsWithoutRef)

**레벨2에서 공용 Input/Button 만들 때 핵심.**

### 문제 상황

공용 `<Button>`을 만들었는데, 사용자가 `aria-label`, `disabled`, `type="submit"` 등 HTML button이 갖는 **모든 속성**을 그대로 쓰고 싶음. 근데 일일이 타입 정의하기 싫음.

### 해결: HTML 요소 props 상속

```ts
type NativeButtonProps = ComponentPropsWithoutRef<"button">;
// → HTMLButtonElement의 모든 props (onClick, disabled, type, aria-*, ...)

interface ButtonProps extends NativeButtonProps {
  variant?: "primary" | "secondary";
  size?: "sm" | "md" | "lg";
}

const Button = ({ variant = "primary", size = "md", ...rest }: ButtonProps) => (
  <button className={`btn btn--${variant} btn--${size}`} {...rest} />
);

// 사용
<Button variant="primary" disabled aria-label="저장" onClick={handleSave}>
  저장
</Button>
```

### ComponentProps vs ComponentPropsWithoutRef vs ComponentPropsWithRef

| 타입 | 특징 | 쓸 곳 |
| --- | --- | --- |
| `ComponentProps<'button'>` | ref 포함/제외 불명확 | ❌ 사용 지양 |
| **`ComponentPropsWithoutRef<'button'>`** | ref 제외 | **기본값** — ref 없는 컴포넌트 |
| `ComponentPropsWithRef<'button'>` | ref 포함 | `forwardRef` 쓰는 컴포넌트 |

### Pick으로 일부만 가져오기

모든 HTML props가 필요하진 않을 때:

```ts
type SelectNativeProps = ComponentPropsWithoutRef<"select">;

interface SelectProps
  extends Pick<SelectNativeProps, "id" | "className" | "disabled"> {
  options: string[];
}
```

→ "`id`, `className`, `disabled`만 select에서 상속, 나머지 HTML 속성은 막음" 같은 **API 좁히기**.

## 6. 제네릭 컴포넌트 (`<T extends ...>`)

**🔴 useForm에서 이미 쓰고 있는 그 패턴.** 근데 함수형 컴포넌트에 붙일 때는 문법이 살짝 까다로워서 짚고 감.

### 기본 형태

```tsx
interface SelectProps<T extends Record<string, string>> {
  options: T;
  selectedOption?: keyof T;
  onChange?: (selected: keyof T) => void;
}

const Select = <T extends Record<string, string>>({
  options,
  selectedOption,
  onChange,
}: SelectProps<T>) => {
  // ...
};
```

### 왜 `extends Record<string, string>`?

- `T`가 객체여야 `keyof T`가 의미 있음
- 값 타입까지 제약 걸어야 안전 (원하는 형태만 받게)

### JSX와 제네릭 화살표 함수 충돌 — `.tsx`에서 주의

```tsx
// ❌ 에러: <T>가 JSX 태그로 파싱됨
const Select = <T>(props: SelectProps<T>) => { ... };

// ✅ 해결 1: extends 추가
const Select = <T extends Record<string, string>>(...) => { ... };

// ✅ 해결 2: 콤마 붙이기 (코드 스타일에 따라)
const Select = <T,>(props: SelectProps<T>) => { ... };

// ✅ 해결 3: function 선언문 쓰기
function Select<T>(props: SelectProps<T>) { ... }
```

**이스타 useForm 케이스**: `<T extends Record<string, string>>` 쓰고 있으니 자동으로 해결됨.

### 사용 시 타입 추론

```tsx
const fruits = { apple: "사과", banana: "바나나" };
type Fruit = keyof typeof fruits;

<Select
  options={fruits}
  onChange={(selected) => {
    // selected는 'apple' | 'banana'로 정확히 추론됨 🎉
  }}
/>
```

제네릭 안 쓰면 `selected: string`밖에 못 받음. 제네릭은 **API 호출자의 타입 정보를 컴포넌트 내부까지 흘려보내는 통로**.

## 7. Pick, Partial 실전 활용

### Pick — "이 props 중 일부만 가져올게"

```ts
interface Restaurant {
  id: number;
  name: string;
  category: RestaurantCategory;
  description?: string;
}

// 리스트 아이템에는 id, name, category만 필요
type RestaurantItemProps = Pick<Restaurant, "id" | "name" | "category">;
```

### Partial — "전부 optional로 바꿔줘"

```ts
interface StyledSelectProps {
  fontSize: "sm" | "md" | "lg";
  color: "primary" | "danger";
}

// 스타일은 기본값이 있어서 optional이어야 함
interface SelectProps extends Partial<StyledSelectProps> {
  options: string[];
}

// → { fontSize?: ..., color?: ..., options: string[] }
```

### Omit — "이건 빼고"

```ts
type RestaurantFormValues = Omit<Restaurant, "id">;
// id 없이 name, category, description만. 새 음식점 생성 시 사용.

// AddRestaurantModal.tsx에서 이미 쓰고 있음!
onAddRestaurant: (newRestaurant: Omit<Restaurant, "id">) => void;
```

**실무**: `Omit<Restaurant, "id">`는 이스타가 이미 쓰고 있는 패턴. `Pick`은 "필요한 것만 고르기", `Omit`은 "안 쓰는 것만 빼기". 개수가 적은 쪽을 쓰면 됨.

## 8. 공변성 vs 반공변성 (맛보기)

**이건 타입 챌린지 고급 영역인데, 8장에서도 나옴. 간단히만.**

### 화살표 함수 prop vs 메서드 표현식

```ts
interface PropsA {
  onChange?: (value: "apple") => void; // 화살표 함수 스타일
}

interface PropsB {
  onChange?(value: "apple"): void; // 메서드 스타일
}
```

겉보기엔 같아 보이지만 **타입 체커의 엄격함이 다름**:

- **화살표**: `strict` 모드에서 **매개변수에 대해 반공변적 체크** (안전)
- **메서드**: 매개변수에 대해 **이변적(bivariant) 체크** (느슨, 옛날 호환성)

### 현실적 결론

- **안전을 원하면 화살표 스타일** (`onChange?: (value: X) => void`)
- 이스타가 쓰고 있는 `onClose: () => void`, `onAddRestaurant: (r: Omit<...>) => void`는 **화살표 스타일**이고, 이게 모던 표준.
- 이 개념은 레벨2에서 타입 에러가 "왜 이건 되고 저건 안 되지?" 싶을 때 떠올릴 수 있으면 충분.

## 🎯 이스타에게 필요한 액션 요약

| 상황 | 이렇게 해 |
| --- | --- |
| 새 컴포넌트 props 타입 | `interface XProps { ... }` + 화살표 함수 스타일 |
| children 받을 때 | `children: ReactNode` 직접 명시 (필수면 required로) |
| React.FC | **쓰지 마** |
| 이벤트 핸들러 | 필요하면 `ChangeEvent<HTML*Element>`, JSX 인라인이면 생략 |
| DOM 전역 리스너 | React 이벤트와 다르니 주의 (KeyboardEvent 등) |
| 공용 Button/Input | `ComponentPropsWithoutRef<'button'>` 상속 + rest spread |
| 공용 컴포넌트에서 유연한 타입 | 제네릭 `<T extends ...>` 패턴 |
| 일부만 떼어오기 | `Pick`, `Omit`, `Partial` 조합 |
| callback 시그니처 | 화살표 스타일 (`(v: X) => void`) |
