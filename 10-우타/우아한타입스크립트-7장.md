## 1. 🔴 왜 API 로직을 컴포넌트에서 분리하는가

### 문제 상황

```tsx
// ❌ 컴포넌트 안에서 직접 fetch
function JobList() {
  const [jobs, setJobs] = useState([]);
  useEffect(() => {
    fetch("/jobs?hardWorking=true", {
      headers: { "API-Version": "v1" }
    })
      .then(res => res.json())
      .then(setJobs);
  }, []);
  return <ul>{jobs.map(/* ... */)}</ul>;
}
```

**뭐가 문제?**
- 요구사항이 바뀔 때마다 (`hardWorking=true` 삭제, API-Version 변경, 헤더에 토큰 추가...) 모든 컴포넌트 수정
- 여러 컴포넌트가 같은 API를 호출하면 로직 중복
- 테스트하기 어려움 (컴포넌트 렌더링과 네트워크 호출이 엉킴)

### 해결: 서비스 레이어 분리

```ts
// services/jobService.ts
export const fetchJobNameList = async (number?: number, size?: number) => {
  const apiRequester = axios.create({
    baseURL: "https://api.baemin.com",
    headers: { "Content-Type": "application/json" }
  });

  return apiRequester.get("/jobs", { params: { number, size } });
};
```

```tsx
// 컴포넌트는 도메인 로직만
function JobList() {
  const [jobs, setJobs] = useState<Job[]>([]);
  useEffect(() => {
    fetchJobNameList().then(res => setJobs(res.data));
  }, []);
  return <ul>{/* ... */}</ul>;
}
```

> **Clean Architecture에서 말하는 "계층 분리".** 이스타가 전에 본 Usecase/Repository 패턴도 같은 맥락.

## 2. axios 인스턴스 다중화

**외부 API마다 baseURL, 헤더, 인터셉터가 다름** → 인스턴스를 여러 개 만듦.

```ts
const apiRequester = axios.create(defaultConfig);

const orderApiRequester = axios.create({
  baseURL: "https://api.baemin.order/",
  ...defaultConfig,
});

const orderCartApiRequester = axios.create({
  baseURL: "https://cart.baemin.order/",
  ...defaultConfig,
});
```

각 인스턴스에 **별도 인터셉터**를 달아 공통 헤더·에러 처리·로깅을 중앙화.

```ts
orderApiRequester.interceptors.request.use(setRequestDefaultHeader);
orderApiRequester.interceptors.response.use(
  res => res,
  httpErrorHandler,
);
```

## 3. 공통 응답 타입과 unknown

### 서버가 내려주는 공통 응답 껍데기

```ts
interface Response<T> {
  data: T;
  status: string;
  serverDateTime: string;
  errorCode?: string;
  errorMessage?: string;
}
```

### 도메인 타입과 조합

```ts
interface JobListItemResponse { name: string }

const fetchJobList = (): AxiosPromise<Response<JobListItemResponse[]>> =>
  apiRequester.get(`/jobs`);
```

### 🔴 Response 타입이 apiRequester 모르게 관리되어야

```ts
// ❌ 잘못된 방식: apiRequester가 Response<T>를 안다
class APIRequester<T> {
  get(url: string): Promise<Response<T>> { /* ... */ }
}

// ✅ 올바른 방식: apiRequester는 unknown 반환, 각 API가 타입 특정
class APIRequester {
  get<T>(url: string): Promise<T> { /* ... */ }
}

const fetchJobList = () =>
  apiRequester.get<Response<JobListItemResponse[]>>("/jobs");
```

> **이유:** 공용 클라이언트가 도메인 타입을 알면 확장성 망함. 미지의 응답은 `unknown`으로 두고 호출 시점에 타입을 씌움.

## 4. 🔴 API 상태 관리 (useState vs 커스텀 훅 vs react-query)

### 단계 1: 컴포넌트 안에서 useState (미션 5단계의 출발점)

```tsx
function RestaurantList() {
  const [data, setData] = useState<Restaurant[]>([]);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState<Error | null>(null);

  useEffect(() => {
    setLoading(true);
    fetchRestaurants()
      .then(setData)
      .catch(setError)
      .finally(() => setLoading(false));
  }, []);

  // 렌더링 분기
}
```

**문제점:**
- 컴포넌트마다 `loading`, `error`, `data` 3종 세트 반복
- 캐시/중복 요청 관리 없음
- stale while revalidate 같은 패턴 직접 구현 필요

### 단계 2: 커스텀 훅으로 추출

```ts
function useFetch<T>(url: string) {
  const [data, setData] = useState<T | null>(null);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState<Error | null>(null);

  useEffect(() => {
    setLoading(true);
    apiRequester.get<T>(url)
      .then(res => setData(res.data))
      .catch(setError)
      .finally(() => setLoading(false));
  }, [url]);

  return { data, loading, error };
}
```

**남는 문제:** 캐시, 리페치, 옵티미스틱 업데이트, 윈도우 포커스 시 재요청 등.

### 단계 3: react-query 등장

```tsx
function RestaurantList() {
  const { data, isLoading, error } = useQuery({
    queryKey: ["restaurants"],
    queryFn: fetchRestaurants,
  });
}
```

**왜 react-query를 쓰나:**
- 서버 상태 = 클라이언트 상태와 다른 **특수한 상태**. Redux/Zustand에 넣으면 스토어가 비대해짐
- 캐시, 무효화, 백그라운드 리페치, 중복 제거가 **기본 제공**
- 로딩/에러 상태를 의식적으로 관리하지 않아도 됨

> **레벨2 순서 권장:** useState → useFetch 훅 → react-query. 직접 만들며 아픔을 느껴봐야 react-query가 "왜" 있는지 체감됨.

## 5. 🔴 에러 핸들링 (isAxiosError + 서브클래싱 + 인터셉터)

### 5-1. 서버 에러 응답 타입 정의

```ts
interface ErrorResponse {
  status: string;
  serverDateTime: string;
  errorCode: string;
  errorMessage: string;
}
```

### 5-2. 타입 가드로 Axios 에러 구분

```ts
function isServerError(error: unknown): error is AxiosError<ErrorResponse> {
  return axios.isAxiosError(error);
}
```

### 5-3. try/catch에서 활용

```ts
const onClickDeleteHistory = async (id: string) => {
  try {
    await apiRequester.post("/order/history/delete", { id });
    alert("삭제되었습니다.");
  } catch (e: unknown) {
    if (isServerError(e) && e.response?.data.errorMessage) {
      setErrorMessage(e.response.data.errorMessage);
      return;
    }
    setErrorMessage("일시적인 에러가 발생했습니다.");
  }
};
```

### 5-4. 에러 서브클래싱 — 도메인별 에러 구분

```ts
class OrderHttpError extends Error {
  private readonly privateResponse: AxiosResponse<ErrorResponse> | undefined;

  constructor(message?: string, response?: AxiosResponse<ErrorResponse>) {
    super(message);
    this.name = "OrderHttpError";
    this.privateResponse = response;
  }

  get response(): AxiosResponse<ErrorResponse> | undefined {
    return this.privateResponse;
  }
}

class AuthError extends Error {
  constructor() {
    super("Unauthorized");
    this.name = "AuthError";
  }
}

class NetworkError extends Error { /* ... */ }
class UnauthorizedError extends Error { /* ... */ }
```

**활용:**
```ts
if (error instanceof AuthError) {
  // 로그인 페이지로 리다이렉트
} else if (error instanceof NetworkError) {
  // 네트워크 재시도 UI
}
```

### 5-5. 인터셉터로 중앙 처리

```ts
apiRequester.interceptors.response.use(
  res => res,
  (error: AxiosError) => {
    if (error.response?.status === 401) {
      window.location.href = "/login";
    }
    return Promise.reject(error);
  }
);
```

> **실무 관용구:** 401/403/500 같은 **공통 처리**는 인터셉터에서, **화면마다 다른 메시지**는 컴포넌트의 try/catch에서.

## 6. Error Boundary + try/catch 역할 분담

### try/catch
- **예상 가능한** 에러 (API 응답 에러, 특정 상태 코드)
- **화면 상의 메시지 표시**가 필요한 경우
- **유저 액션 복구 가능** (재시도, 폼 수정)

### Error Boundary
- **예상 못한** 런타임 에러 (컴포넌트 내부 throw, 렌더 에러)
- **컴포넌트 트리 전체**의 폴백 UI 표시
- **Sentry 등으로 리포팅**

```tsx
class ErrorBoundary extends React.Component {
  state = { hasError: false };

  static getDerivedStateFromError() {
    return { hasError: true };
  }

  componentDidCatch(error: Error, info: React.ErrorInfo) {
    // Sentry.captureException(error);
  }

  render() {
    if (this.state.hasError) return <ErrorFallback />;
    return this.props.children;
  }
}
```

> **react-query를 쓰면** `useQueryErrorResetBoundary`로 Error Boundary에서 "다시 시도" 버튼까지 자연스럽게 연결 가능.

## 7. 🔴 Superstruct로 런타임 타입 검증

### 왜 필요한가 (6장과 연결)

**TS 타입은 런타임에 사라짐.** 서버가 약속과 다른 데이터를 내려주면 TS는 모른다.

```ts
interface ListItem { id: string; content: string }

const res = await apiRequester.get<ListItem[]>("/items");
// 서버가 [{ id: 1 }] (숫자) 내려줘도 TS는 ListItem[]이라 믿음
res.data[0].content.toUpperCase();  // 🚨 런타임 TypeError
```

### Superstruct로 런타임 검증

```ts
import { object, string, assert, is, validate } from "superstruct";

const ListItemSchema = object({
  id: string(),
  content: string(),
});

// 세 가지 방식
assert(data, ListItemSchema);    // 불일치 시 throw
is(data, ListItemSchema);        // boolean 반환 (타입 가드로 작동)
validate(data, ListItemSchema);  // [error, data] 튜플

// 실전
const fetchListItems = async () => {
  const res = await apiRequester.get<unknown>("/items");
  if (!is(res.data, array(ListItemSchema))) {
    throw new Error("응답 형식이 예상과 다릅니다.");
  }
  return res.data;  // 여기선 ListItem[]로 좁혀짐
};
```

> **대안 라이브러리:** Zod (더 유명), io-ts, Yup. 요즘 신규 프로젝트는 Zod 선호.

## 8. API 모킹 전략

개발 단계에서 서버 없이 작업하는 방법들.

| 방법 | 특징 |
| --- | --- |
| **JSON 파일 불러오기** | 가장 단순. `import data from './mock.json'`. 정적이라 CRUD 시뮬레이션은 안 됨 |
| **JSON Server** | 파일을 서버로 띄움. 레벨2 미션 5단계에서 쓰는 방식 |
| **Next.js API Routes** | 프레임워크 내에서 핸들러 작성 |
| **axios-mock-adapter** | axios 요청을 가로채서 가짜 응답 반환. 환경변수로 on/off |
| **MSW (Mock Service Worker)** | Service Worker로 네트워크 레벨에서 가로챔. 실무 표준 |

### 환경변수로 모킹 제어

```ts
if (process.env.REACT_APP_MOCK === "true") {
  mock.onGet("/jobs").reply(200, jobListMock);
}
```

같은 코드베이스에서 로컬/서버를 유연하게 전환.

| --- | --- |
