## 1. 타입 확장하기

**타입 확장 = 기존 타입으로 새 타입을 정의하는 것.** 장점 3가지:

1. 중복 제거
2. 명시적 상속 관계 드러내기 (`BaseCardItem extends BaseMenuItem`을 읽으면 구조가 바로 보임)
3. 확장성 (Base만 고치면 하위 타입 전부 반영)

### 배민 장바구니 예시

```ts
interface BaseMenuItem {
  itemName: string | null;
  itemImageUrl: string | null;
  itemDiscountAmount: number;
  stock: number | null;
}

interface BaseCardItem extends BaseMenuItem {
  quantity: number;  // ← 메뉴 타입을 그대로 상속받고 quantity만 추가
}

interface EditableCardItem extends BaseCardItem {
  isSoldOut: boolean;
  optionGroups: SelectableOptionGroup[];
}

interface EventCardItem extends BaseCardItem {
  orderable: boolean;
}
```

## 2. `type` vs `interface` 확장 문법

| 키워드 | 확장 방법 |
| --- | --- |
| `interface` | `interface B extends A {}` |
| `type` | `type B = A & { ... }` (교차 타입) |

```ts
// interface 버전
interface BaseCardItem extends BaseMenuItem {
  quantity: number;
}

// type 버전
type BaseCardItem = BaseMenuItem & {
  quantity: number;
};
```

**실무 관용구** (이스타가 3장에서 본 것):
- 재선언 병합(declaration merging)이 필요하면 `interface`
- 유니온/조건부/매핑된 타입이 필요하면 `type`
- 헷갈리면 `type`으로 통일해도 됨

## 3. 유니온 vs 교차 — "또는" vs "그리고"

이스타가 자주 헷갈려하는 구조적 차이.

### 유니온 (`|`) — "또는"

```ts
type CookingStep = { orderId: string; price: number };
type DeliveryStep = { orderId: string; time: number; distance: string };

type Step = CookingStep | DeliveryStep;

function getDistance(step: Step) {
  return step.distance;  // 🚨 Property 'distance' does not exist on type 'CookingStep'
}
```

유니온 타입으로 `Step`을 선언하면 **두 타입의 공통 속성(`orderId`)에만 접근 가능**. 각 타입의 고유 속성에 접근하려면 타입 좁히기가 필요.

### 교차 (`&`) — "그리고"

```ts
type BaedalProgress = CookingStep & DeliveryStep;
// { orderId: string; price: number; time: number; distance: string }

function getOrder(progress: BaedalProgress) {
  return `${progress.price} / ${progress.distance}`;  // ✅ 둘 다 접근 가능
}
```

교차 타입은 **두 타입의 모든 속성을 다 가져야 함**.

> **주의:** 서로 호환되지 않는 타입끼리 교차하면 `never`가 됨. `type X = number & string` → `never`.

## 4. 타입 좁히기 (Type Narrowing)

**좁히기 = 변수 타입을 더 구체적인 타입으로 좁혀가는 과정.** 런타임 체크 코드가 컴파일타임 타입 정보도 좁힘.

### typeof 연산자

```ts
function getDate(date: string | Date) {
  if (typeof date === "string") {
    return date.toUpperCase();  // ✅ date: string
  }
  return date.getTime();  // ✅ date: Date
}
```

**주의:** `typeof`는 JS 원시 타입만 구분 (`string`, `number`, `boolean`, `undefined`, `object`, `function`, `symbol`, `bigint`). 배열·객체·클래스 인스턴스는 전부 `"object"`라 구분 못 함.

### instanceof

클래스 인스턴스를 좁힐 때.

```ts
function handleErr(err: Error | string) {
  if (err instanceof TypeError) {
    console.log(err.message);  // ✅ TypeError
  }
}
```

### in 연산자

객체에 특정 프로퍼티가 있는지로 좁히기.

```ts
function handle(step: CookingStep | DeliveryStep) {
  if ("distance" in step) {
    return step.distance;  // ✅ DeliveryStep
  }
  return step.price;  // ✅ CookingStep
}
```

> 옵셔널 속성(`distance?: string`)도 `in`으로 좁힐 수 있음. 다만 이때는 **discriminated union**(아래 6번)이 더 안전.

## 5. 🔴 사용자 정의 타입 가드 `is`

**타입스크립트가 자동으로 좁혀주지 못하는 경우를 위한 수동 가드.**

### 문법

```ts
function isString(value: unknown): value is string {
  return typeof value === "string";
}
```

반환 타입이 `value is string`인 함수가 `true`를 리턴하면, 호출한 쪽에서 `value`는 `string`으로 좁혀짐.

### 배민 실전 예시 — 옵셔널 props에서 런타임 검증 + 타입 가드 동시에

```ts
const isDestinationCode = (x: string): x is DestinationCode => {
  return destinationCodeList.includes(x);
};

const getAvailableDestinationNameList = async (): Promise<DestinationName[]> => {
  const data = await AsyncDestinationCodeApi();
  const result: DestinationName[] = [];

  data.forEach((codeItem) => {
    if (isDestinationCode(codeItem.destination)) {
      // ✅ 여기서 codeItem.destination은 DestinationCode로 좁혀짐
      result.push(DestinationCodeToNameMap[codeItem.destination]);
    }
  });

  return result;
};
```

서버가 내려준 `string`을 믿을 수 없으니 런타임에 배열 포함 여부로 검증 + TS에게 "이제 이건 `DestinationCode`다"라고 알려줌.

### 언제 쓰나
- `unknown` / `string` 같은 넓은 타입에서 도메인 타입으로 좁힐 때
- 자체 검증 로직이 필요한 곳 (API 응답, 사용자 입력)
- `as` 캐스팅하고 싶어질 때 — 대신 **타입 가드 함수**를 만들어 런타임 체크까지 같이 하는 게 안전

## 6. 식별할 수 있는 유니온 (Discriminated Union)

**공통 "식별자" 필드로 분기하는 유니온.** 옵셔널 속성 + `in` 조합보다 안전.

### Before (옵셔널이라 구멍이 생김)

```ts
type TextError = { errorCode: string; errorMessage: string };
type ToastError = { errorCode: string; errorMessage: string; toastShowDuration: number };
type AlertError = { errorCode: string; errorMessage: string; onConfirm: () => void };

type ErrorFeedbackType = TextError | ToastError | AlertError;

const errorArr: ErrorFeedbackType[] = [
  // 🚨 toastShowDuration, onConfirm 둘 다 있어도 에러 안 남 (ToastError & AlertError로 추론됨)
  { errorCode: "500", errorMessage: "server error", toastShowDuration: 3000, onConfirm: () => {} },
];
```

### After (discriminator 추가)

```ts
type TextError = { errorType: "TEXT"; errorCode: string; errorMessage: string };
type ToastError = { errorType: "TOAST"; errorCode: string; errorMessage: string; toastShowDuration: number };
type AlertError = { errorType: "ALERT"; errorCode: string; errorMessage: string; onConfirm: () => void };

type ErrorFeedbackType = TextError | ToastError | AlertError;

const errorArr: ErrorFeedbackType[] = [
  // 🚨 ERROR: errorType이 "TOAST"인데 onConfirm이 있으면 안 됨
  { errorType: "TOAST", errorCode: "500", errorMessage: "...", toastShowDuration: 3000, onConfirm: () => {} },
];
```

`errorType`이라는 **리터럴 유니온 필드**로 구분. `switch (err.errorType)` 분기가 완벽히 좁혀짐.

> **React에서:** `useReducer`의 action 타입을 이렇게 설계함. `type Action = { type: "ADD"; payload: ... } | { type: "REMOVE"; id: string }`.

## 7. 🔴🔴 Exhaustiveness Checking — never 로 모든 케이스 강제

**"모든 케이스를 빠짐없이 처리했는지" 컴파일러가 체크하게 만드는 패턴.** 리터럴 유니온이 커질 때 반드시 알아야 함.

### 문제 상황

```ts
type ProductPrice = "10000" | "20000";

const getProductName = (price: ProductPrice): string => {
  if (price === "10000") return "배민상품권 1만 원";
  if (price === "20000") return "배민상품권 2만 원";
  return "배민상품권";  // fallback
};
```

나중에 타입에 `"5000"`이 추가되면?

```ts
type ProductPrice = "10000" | "20000" | "5000";  // ← 타입만 추가됨
// getProductName은 수정 안 해도 에러 안 남. 런타임에 "배민상품권"으로 fallback.
```

**타입 변경 → 사용처 강제 업데이트가 안 됨 = 유지보수 지옥.**

### 해결: `never` 파라미터 헬퍼

```ts
const exhaustiveCheck = (x: never): never => {
  throw new Error(`Unhandled case: ${x}`);
};

const getProductName = (price: ProductPrice): string => {
  if (price === "10000") return "배민상품권 1만 원";
  if (price === "20000") return "배민상품권 2만 원";
  exhaustiveCheck(price);
  // 🚨 Argument of type '"5000"' is not assignable to parameter of type 'never'
};
```

### 작동 원리

1. `if` 분기를 타고 가면 TS가 `price`의 가능성을 하나씩 제거
2. 모든 분기를 통과하면 `price`는 `never`로 좁혀짐
3. `exhaustiveCheck(price: never)`에 넘길 수 있음
4. 하지만 `"5000"`이 남아 있으면 `never`로 좁혀지지 않음 → **타입 에러**

### 언제 쓰나
- **리터럴 유니온 타입에 대해 분기할 때** (switch, if-else 체인)
- **useReducer의 action.type switch**
- **카테고리/상태머신 (`"pending" | "success" | "error"`)**

> **핵심:** 타입을 늘리면 에러가 터지는 게 좋음. 조용한 fallback은 나중에 디버깅으로 돌아옴.
