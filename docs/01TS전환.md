## Phase A-2 — 첫 과제: TS 셋업

### 🎯 목적

React 코드에 타입을 붙이려면 **TS 컴파일러가 JSX 를 이해해야** 함. 그 설정을 먼저 깔아요.

### 📚 왜 이 세 가지가 다 필요한가

1. **`typescript` 패키지** — 컴파일러 본체. 없으면 `.tsx` 를 해석 못 함.
2. **`tsconfig.json`** — "이 프로젝트의 TS 규칙" 을 명시.
    - 특히 `"jsx": "react-jsx"` 옵션이 핵심. 이게 있어야 TS 가 JSX 를 자바스크립트로 변환할 줄 알고, 더 이상 `import React from "react"` 를 파일마다 쓰지 않아도 돼요. (React 17+ 방식)
    - `"strict": true` 를 켜두면 타입 챌린지처럼 빡빡한 모드. 이스타 수준이면 당연히 켜고 갈 만함.
3. **`src/declarations.d.ts`** — `import "./style.css"` 같은 코드를 TS 가 에러로 잡지 않도록 "CSS 는 타입이 있는 모듈이라고 간주해라" 고 알려주는 선언 파일.
    - 이거 없으면 `App.tsx` 에서 `import "./style.css"` 하자마자 빨간 줄 떠요.

### 📝 과제

아래 두 가지를 순서대로 해주세요.

**1. typescript 패키지 설치**

`npm install -D typescript`

**2. `tsconfig.json` 을 프로젝트 루트에 생성**

Vite + React 18 에 맞는 설정. 내용은 아래 그대로:
```json
{
  "compilerOptions": {
    "target": "ES2020",
    "useDefineForClassFields": true,
    "lib": ["ES2020", "DOM", "DOM.Iterable"],
    "module": "ESNext",
    "skipLibCheck": true,
    "moduleResolution": "bundler",
    "allowImportingTsExtensions": true,
    "resolveJsonModule": true,
    "isolatedModules": true,
    "noEmit": true,
    "jsx": "react-jsx",
    "strict": true,
    "allowArbitraryExtensions": true
  },
  "include": ["src"]
}
```

**3. `src/declarations.d.ts` 파일을 만들고 한 줄만 적기**

```ts
declare module "*.css";
```

## Phase A-2 — 3번째 과제: 이미지 import 에러 걷어내기 (유형 ①)

### 📝 과제

`src/declarations.d.ts` 를 열어서 이미지 관련 선언을 추가해주세요.

이렇게 써도 되고:

`declare module "*.png";`

조금 더 정확히 쓰고 싶으면 (Vite 가 실제로는 URL 문자열을 반환하니까):

`declare module "*.png" {
  const src: string;
  export default src;
}`

둘 다 에러는 없어지는데, 후자는 **`import addButton from "..."` 의 `addButton` 이 `string` 타입으로 정확히 추론** 돼요.

> 💡 Vite 를 쓰면 `vite/client` 라는 공식 타입도 있어요 (`/// <reference types="vite/client" />`). 이걸 `declarations.d.ts` 맨 위에 넣으면 PNG/JPG/SVG/환경변수 등이 전부 한 방에 해결돼요.

## Phase A-2 — 4번째 과제: 공통 데이터 타입 정의

### 🎯 목적

남은 16개 에러를 보면 대부분 **"Restaurant 가 뭔지, Category 가 뭔지 TS 가 모른다"** 로 귀결돼요. 먼저 **도메인의 핵심 타입부터 정의**해두면, 그 뒤 props 타입 붙이기가 훨씬 수월해져요.

### 📚 왜 도메인 타입을 먼저?

두 가지 선택지가 있어요:

**A. 컴포넌트마다 props 인터페이스를 각자 정의**

`// RestaurantList
interface Props {
  restaurants: { id: string; name: string; category: string; ... }[];
}

// RestaurantDetailModal
interface Props {
  restaurant: { id: string; name: string; category: string; ... };
}`

→ **같은 모양이 3-4 곳에 중복**. `category` 를 `string` 으로 느슨하게 쓸지 유니온으로 좁힐지도 파일마다 엇갈림.

**B. `Restaurant`, `Category` 를 먼저 정의하고 재사용**

`interface Props {
  restaurants: Restaurant[];
}`

→ 한 군데 바꾸면 전부 반영. 도메인이 코드에 **한 번만** 나타남.

B 가 실무 표준. 이게 바로 **"도메인 타입을 공용 위치에 모은다"** 는 패턴이에요.

### 📚 한 가지 설계 판단

`Category` 를 어떻게 정의할지 고민거리가 있어요. 현재 `db.json` 의 category 값은 **한식/중식/일식/양식/아시안/기타** 중 하나.

그런데 App.tsx:20 의 `const [category, setCategory] = useState("전체");` 를 보면 **"전체"** 도 상태값이에요. 하지만 "전체" 는 **음식점의 속성** 이 아니라 **필터 UI 의 선택지**.

두 개념을 분리할 수 있어요:

`// 음식점이 가질 수 있는 카테고리
type RestaurantCategory = "한식" | "중식" | "일식" | "양식" | "아시안" | "기타";

// 필터가 가질 수 있는 선택지 ("전체" 포함)
type FilterCategory = "전체" | RestaurantCategory;`

이렇게 하면 **`Restaurant.category` 에는 "전체" 가 절대 들어갈 수 없고**, 필터 state 는 둘 다 허용. 타입 챌린지 푼 이스타 수준이면 이 분리 감각이 중요해요.

### 📝 과제

**1. 타입 파일 위치 결정**

고민: 작은 프로젝트는 `src/types.ts` 하나에 몰아도 OK. 도메인이 여러 개면 `src/types/restaurant.ts` 같이 쪼갬. **이 프로젝트는 작으니 `src/types.ts` 한 파일로 충분.**

**2. `src/types.ts` 생성 후 아래 두 타입 정의**

- `RestaurantCategory` — 리터럴 유니온
- `FilterCategory` — "전체" + RestaurantCategory
- `Restaurant` — id, name, category, description 를 가진 인터페이스

`db.json` 을 열어서 실제 데이터 모양을 먼저 확인하고 정의하세요. 타입은 **실제 데이터와 정확히 일치** 해야 해요 (optional 필드 여부 포함).

**3. App.tsx 의 `useState` 에 적용**

App.tsx:12 의 `useState([])` 는 지금 TS 가 `never[]` 로 추론하고 있어요 (빈 배열에서 타입 정보 없음). 여기에 제네릭으로 `Restaurant[]` 를 명시:

`const [restaurants, setRestaurants] = useState<Restaurant[]>([]);`

App.tsx:20 의 category state 도 `FilterCategory` 타입으로 명시.

**4. 확인**

`npx tsc --noEmit 2>&1 | head -30`

이걸 돌려서 **TS2339 (category does not exist on never) 에러가 사라졌는지** 확인. 아직 다른 에러는 남아있는 게 정상.

<details> <summary>interface vs type — 뭘 쓸까</summary>

- **객체 모양 = `interface`** (extends 가능, 선언 병합 등 확장성)
- **유니온/리터럴 = `type`** (interface 는 유니온 못 만듦)

그래서 `Restaurant` 는 `interface`, `RestaurantCategory` 는 `type` 이 자연스러움.

</details>
