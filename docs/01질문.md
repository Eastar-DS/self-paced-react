# TS 세팅

## 📂 생성한 파일별 역할

### 1. `typescript` 패키지 (devDependencies)

- **정체:** TS 컴파일러 / 타입체커 본체. `npx tsc` 명령어가 이 패키지를 호출함.
- **Vite 와의 관계:** Vite 는 `.tsx` 를 빌드는 해주지만 **타입 검사는 안 함** (속도 때문에 esbuild 가 타입만 벗겨냄). 실제 타입 검사는 **VS Code 의 TS 서버** 와 **`tsc --noEmit`** (CI 용) 가 담당.
- **그래서 역할 분리:**
  - Vite (esbuild) → `.tsx` → `.js` 변환 + 번들링
  - TypeScript (`tsc`) → 타입 검사만
  - 둘은 서로 독립적으로 돌아감.

### 2. `tsconfig.json`

- **정체:** "이 프로젝트에서 TS 를 어떻게 쓸지" 를 정의하는 설정 파일.
- **누가 읽나:**
  - VS Code 의 TS 언어 서버 (빨간 줄 표시)
  - `tsc` 명령어
  - ESLint 의 TS 플러그인
- **특징:** 파일 하나가 프로젝트 전체의 타입 엄격도를 좌우. 팀 컨벤션의 핵심 문서.

### 3. `src/declarations.d.ts`

- **정체:** **"모듈 선언 파일"** 이라 부름. `.d.ts` 확장자 = "타입 선언만 담은 파일" (실행되는 코드 없음).
- **이 파일의 내용 `declare module "*.css";` 의미:**
  - "`*.css` 패턴에 매칭되는 모든 import 는 유효한 모듈이라고 치자"
  - export 되는 구체적 값/타입은 없지만, **import 문 자체는 허용**.
  - 없으면 `import "./style.css"` 에서 `TS2307: Cannot find module './style.css'` 에러.
- **확장 가능:** 나중에 CSS Modules (`import styles from "./X.module.css"`) 쓰면 이렇게 바꿈:
  ```ts
  declare module "*.module.css" {
    const classes: { readonly [key: string]: string };
    export default classes;
  }
  declare module "*.css";
  ```

## ⚙️ `tsconfig.json` 옵션별 의미

### `target: "ES2020"`

컴파일 결과물의 JS 버전. ES2020 이면 optional chaining (`?.`), nullish coalescing (`??`), `Promise.allSettled` 등 문법/API 를 **변환하지 않고 그대로 유지**. 구형 브라우저 지원 안 하면 OK.

실전: Vite 는 실제로 자체 esbuild 로 이 옵션과 별개로 변환함. `tsconfig.json` 의 `target` 은 "`tsc` 가 검사할 때 기준" 정도로 봐도 됨.

### `useDefineForClassFields: true`

클래스 필드를 ECMAScript 표준(`Object.defineProperty`) 방식으로 처리. TC39 표준에 맞춘 동작. 레거시 (`this.x = ...` 할당 방식) 행동 방지.

실전: 로또 코드의 `class Lotto { #numbers = []; }` 같은 클래스가 표준 대로 동작.

### `lib: ["ES2020", "DOM", "DOM.Iterable"]`

**이 환경에서 사용 가능한 글로벌 타입** 을 명시.

- `ES2020` — ES2020 표준에 포함된 `Array`, `Promise`, `String.prototype.matchAll` 등의 타입.
- `DOM` — `document`, `window`, `HTMLElement`, `HTMLInputElement` 등 브라우저 DOM 타입. 없으면 React 의 DOM 이벤트 타입이 깨짐.
- `DOM.Iterable` — `document.querySelectorAll(...)` 결과를 `for...of` 로 순회 가능하게 하는 iterator 타입 추가.

### `module: "ESNext"`

출력할 모듈 시스템 = 최신 ESM (`import`/`export`). Vite 번들러가 최종 처리 맡으니 TS 는 변환 안 하고 유지.

### `skipLibCheck: true`

`node_modules` 안의 `.d.ts` 파일 **타입 검사 생략**.

- **왜 켜나:** 빌드 속도 대폭 향상 + 남의 라이브러리 타입 버그에 방해받지 않음.
- **대부분 프로젝트에서 켜는 게 표준**. TS 팀도 공식 권장.

### `moduleResolution: "bundler"`

모듈 경로 해석 방식. `"bundler"` 는 Vite/Webpack/esbuild 같은 **번들러의 동작 방식** 을 따라감.

- 이전 표준: `"node"` → Node.js 의 CommonJS 해석 방식. 번들러 환경과 살짝 어긋나는 케이스가 있었음.
- TS 5+ 부터 권장: `"bundler"` → 실제 번들러가 하는 대로 해석. `.ts` 확장자 생략, package.json 의 `exports` 필드 정확 해석 등.

### `allowImportingTsExtensions: true`

`import foo from "./foo.ts"` 처럼 **`.ts` 확장자를 명시한 import** 허용.

- 보통은 확장자 생략(`from "./foo"`)하지만, Deno 등 환경에선 확장자 필수.
- `moduleResolution: "bundler"` + `noEmit: true` 조합에서만 가능.

### `resolveJsonModule: true`

`import data from "./data.json"` 처럼 JSON 파일을 **모듈로 import** 허용. 타입도 JSON 구조대로 자동 추론됨.

실전: 설정 파일, 더미 데이터 넣을 때 유용. 로또 상수 중 JSON 쓰는 게 있으면 여기 활용.

### `isolatedModules: true`

**각 파일을 독립적으로** 트랜스파일 가능하도록 강제하는 제약.

- **왜 필요:** Vite (esbuild), SWC 같은 빠른 트랜스파일러는 **파일 단위로** 작업함. 타입 정보 전역 분석 없이.
- **그래서 금지되는 패턴:**
  - `export { Foo }` — Foo 가 타입인지 값인지 파일만 보고 판단 불가. → `export type { Foo }` 로 써야 함.
  - `const enum` — 파일 간 인라인이 필요한데 불가능. → 일반 `enum` 쓰거나 리터럴 유니온으로 대체.
- **TS 챌린지 경험자에겐 새로 익혀야 할 부분**: `import type`, `export type` 문법 습관화.

### `noEmit: true`

**TS 컴파일러가 타입 검사만 하고 JS 파일은 출력 안 함.**

- 실제 JS 변환은 Vite (esbuild) 담당.
- `tsc` 는 검사 전담.
- 역할 분리의 핵심 옵션.

### `jsx: "react-jsx"`

JSX 를 JS 로 변환하는 방식. **React 17+ 의 새 JSX transform**.

- **이게 있어서 좋은 점:** 파일마다 `import React from "react"` 안 써도 됨. `<div>` 만 써도 자동으로 `react/jsx-runtime` 을 호출함.
- 다른 옵션: `"react"` (레거시, React import 필수), `"preserve"` (Babel 등 외부에서 처리).

### `strict: true`

**모든 엄격 모드 옵션 한 번에 켬.** 포함되는 것들:

- `strictNullChecks` — `null`/`undefined` 를 다른 타입과 구분. `string | null` 이면 null 체크 필수.
- `noImplicitAny` — 매개변수 타입 생략 시 암묵적 `any` 금지. 명시 강제.
- `strictFunctionTypes` — 함수 매개변수의 bivariance 방지 (타입 안전성 ↑).
- `strictBindCallApply` — `bind`, `call`, `apply` 인자 타입 검사.
- `strictPropertyInitialization` — 클래스 필드 초기화 강제.
- `alwaysStrict` — 출력 JS 에 `"use strict"` 자동 추가.
- `noImplicitThis` — `this` 가 암묵적 any 면 에러.
- `useUnknownInCatchVariables` — `catch (e)` 의 `e` 를 `any` 가 아닌 `unknown` 으로.

> 이스타가 타입 챌린지 10번까지 푼 수준이면 **당연히 켜고 가야 할 기본값**.

### `allowArbitraryExtensions: true`

`.module.css`, `.svg`, `.png` 같이 **"TS 가 모르는 확장자"** import 를 허용. (단, `declarations.d.ts` 에 선언이 있어야 실제로 통과.)

TS 5+ 의 옵션. Vite 에서 다양한 asset import 할 때 필요.

### `include: ["src"]`

TS 가 검사할 대상 폴더 한정. `node_modules`, `dist` 는 자동 제외되므로 신경 쓸 필요 X.

## 🖼️ `declare module "*.png" { const src: string; export default src; }` 의 뜻

### 문법 뜯어보기

```ts
declare module "*.png" {
  const src: string;
  export default src;
}
```

한 줄씩 해석:

- **`declare module "*.png"`** — "확장자 `.png` 로 끝나는 **모든 경로의 import** 를 여기 정의된 규칙으로 취급해라."
  - `"*.png"` 는 와일드카드 패턴. `./a.png`, `../assets/foo.png` 모두 매칭.
  - `declare` 는 "타입만 선언하고 실제 코드는 없다" 는 키워드.

- **`const src: string;`** — "이 모듈 내부에는 `src` 라는 이름의 `string` 값이 있다" 고 TS 에게 약속.
  - 이름은 `src` 일 필요 없음. `url`, `path`, `whatever` 다 OK. **default export 로 나가는 `string` 이라는 점만 중요.**
  - 왜 `string` 인가: Vite 가 `.png` import 를 **"번들링된 이미지의 URL 문자열"** 로 변환하기 때문. 예: `/assets/add-button-abc123.png`

- **`export default src;`** — "그 `src` 를 default export 로 내보낸다."
  - 이게 있어야 `import addButton from "./add-button.png"` 의 `addButton` 이 **`string` 타입으로 추론** 됨.

### 간단 버전과의 차이

```ts
// 간단 버전
declare module "*.png";

// 풀 버전
declare module "*.png" {
  const src: string;
  export default src;
}
```

| 항목 | 간단 버전 | 풀 버전 |
|---|---|---|
| 에러 사라짐? | ✅ | ✅ |
| `import addButton from "..."` 에서 `addButton` 타입 | `any` | `string` |
| 다른 변수에 할당 시 타입 안전성 | ❌ (any 는 뭐든 허용) | ✅ |
| autocomplete | 없음 | `.length`, `.slice()` 등 string 메서드 |

**실전에선 풀 버전이 권장**. "어차피 URL 문자열" 이라는 정보가 타입에도 남아야 다른 코드에서 안전하게 씀.

### 실제 변환 과정 체감하기

`import addButton from "../assets/images/add-button.png"` 를 썼을 때 Vite 가 하는 일:

1. **개발 모드**: 이미지를 그대로 제공하고 URL 경로 리턴 → `addButton === "/src/assets/images/add-button.png"`
2. **빌드 모드**: 이미지를 `dist/assets/` 로 옮기고 **해시 붙은 파일명** 생성 → `addButton === "/assets/add-button-a3f92b1c.png"`

두 경우 다 **문자열**. 그래서 타입을 `string` 으로 선언한 거예요.

## 🚀 실무에선 `vite/client` 로 한 방에

지금 우리가 직접 `.png`, `.jpg`, `.svg` 등을 하나씩 선언하는 건 **학습용**. 실무에선 Vite 공식 타입 패키지 하나면 끝나요.

### 사용법

`src/declarations.d.ts` (혹은 `src/vite-env.d.ts`) 맨 위에 한 줄:

```ts
/// <reference types="vite/client" />
```

끝. 이 한 줄이 **자동으로 선언해주는 것들**:

- **모든 이미지 확장자**: `.png`, `.jpg`, `.jpeg`, `.gif`, `.svg`, `.ico`, `.webp`, `.avif` …
- **미디어**: `.mp4`, `.webm`, `.ogg`, `.mp3`, `.wav`, `.flac`, `.aac` …
- **폰트/기타 asset**: `.woff`, `.woff2`, `.eot`, `.ttf`, `.otf` …
- **CSS/SCSS 모듈**: `.module.css`, `.module.scss` (타입된 객체 export)
- **URL 쿼리 suffix**: `import raw from "./file?raw"`, `?url`, `?inline` 등
- **환경변수**: `import.meta.env.VITE_API_KEY` 같은 사용자 정의 env 타입

### 공식 소스 위치

`node_modules/vite/client.d.ts` — 열어보면 위에 나열한 것들이 이미 전부 `declare module` 로 적혀 있음. 우리가 지금 수동으로 쓴 거랑 똑같은 문법.

### 보통 프로젝트 구성

Vite 가 `npm create vite@latest` 로 TS 템플릿 만들면 기본으로 이런 파일이 생김:

```ts
// src/vite-env.d.ts
/// <reference types="vite/client" />
```

이게 전부. 프로젝트 시작하자마자 자산 import 걱정 제로.

### 우리 프로젝트의 선택

이 프로젝트는 **수동 선언 유지**. 이유:

1. **학습 효과**: `declare module` 문법을 손으로 써봐야 원리를 익힘. 챌린지 경험자라도 "Vite 가 뭘 해주는지" 감각은 직접 짜봐야 생김.
2. **의존성 축소**: `vite/client` 는 환경변수 등 불필요한 전역 타입도 같이 끌고 옴. 지금 단계에선 오히려 방해.
3. **디버깅 근육**: 나중에 로또 프로젝트 셋업할 땐 `vite/client` 한 줄로 끝낼 거지만, 지금 손으로 해봐야 "그 한 줄이 뭘 대신해주는지" 를 이해함.

### 그래서 언제 `vite/client` 로 갈아타나?

**Phase B (로또 React+TS 전환) 에서.**
로또는 처음부터 TS 로 짤 거고, 실무 표준 셋업을 익히는 게 목적이니까 거기선 `vite/client` 로 시작해요.
