# 04. 목록과 테이블

## 🤔 왜 "목록" 이 따로 있어야 할까?

"음식점 6곳의 이름과 설명" 을 페이지에 늘어놓는다고 해봅시다.
그냥 `<p>` 여섯 개로 쓰면 될 것 같지만, 생각해 보면 이건 **한 묶음의 같은 종류 데이터** 입니다.
HTML 은 이런 **"같은 종류의 항목이 여러 개"** 라는 의미를 표현하는 태그를 따로 줍니다.
바로 목록(list) 이에요.

왜 의미를 구분해야 하냐면:

- 스크린 리더가 "목록 시작. 총 6개 항목." 이라고 미리 알려 줄 수 있습니다.
- CSS 로 "목록" 에만 일괄 스타일을 주기 쉬워집니다.
- **React 에서 `.map()` 으로 반복 렌더링할 때 자연스럽게 매핑** 됩니다. (이게 나중에 중요해요)

---

## 📋 순서 없는 목록: `<ul>` + `<li>`

`<ul>` = Unordered List. 순서가 의미 없는 목록일 때 씁니다.
`<li>` = List Item. 목록의 각 항목입니다.

```html
<ul>
  <li>피양콩할마니</li>
  <li>친친</li>
  <li>잇쇼우</li>
</ul>
```

브라우저가 기본으로 각 `<li>` 앞에 **동그라미 불릿(•)** 을 찍고, 약간의 들여쓰기를 넣습니다.
**이 모양은 CSS 로 없애거나 바꿀 수 있어요.** 실무에서는 디자인에 맞게 바꾸는 경우가 많습니다.

### 규칙

- `<ul>` 의 **직계 자식은 `<li>` 만** 올 수 있습니다. `<ul><p>...</p></ul>` 은 잘못된 HTML.
- `<li>` 안에는 **아무 태그나** 넣어도 됩니다. 또 다른 `<ul>` 을 넣으면 **중첩 목록** 이 됩니다.

```html
<ul>
  <li>
    한식
    <ul>
      <li>피양콩할마니</li>
      <li>친친이 아니라 한식집</li>
    </ul>
  </li>
  <li>중식</li>
</ul>
```

---

## 🔢 순서 있는 목록: `<ol>` + `<li>`

`<ol>` = Ordered List. **순서 자체에 의미가 있을 때** 씁니다.

```html
<ol>
  <li>계란을 푼다</li>
  <li>팬에 기름을 두른다</li>
  <li>계란을 부어 굽는다</li>
</ol>
```

브라우저가 자동으로 `1.`, `2.`, `3.` 번호를 매겨 줍니다.

### `<ul>` vs `<ol>` 고르는 기준

| 질문 | 답 |
|---|---|
| 순서가 바뀌면 의미가 깨지나? | **예** → `<ol>` |
| 순서는 그냥 배치일 뿐, 바꿔도 상관없나? | **예** → `<ul>` |

예:
- 조리 순서 → `<ol>` (순서가 깨지면 음식이 망함)
- 음식점 목록 → `<ul>` (순서 자체엔 의미 없음, 그냥 여러 개)

## 🎯 [templates/index.html](../templates/index.html) 에서 찾아 보기

```html
<ul class="restaurant-list">
  <li class="restaurant">
    <div class="restaurant__category">
      <img src="./category-korean.png" alt="한식" class="category-icon">
    </div>
    <div class="restaurant__info">
      <h3 class="restaurant__name text-subtitle">피양콩할마니</h3>
      <p class="restaurant__description text-body">평양 출신의...</p>
    </div>
  </li>
  <!-- 음식점 수만큼 <li> 반복 -->
</ul>
```

- `<ul>` 로 **"음식점 목록"** 임을 선언
- 각 음식점이 하나의 `<li>` 가 됨
- `<li>` 안에는 이미지와 설명이 섞여 있지만, 전체가 **"하나의 항목"** 이라는 의미는 유지됨

> 🔗 **React 로 가면**:
> ```jsx
> <ul>
>   {restaurants.map(r => <li key={r.id}>{r.name}</li>)}
> </ul>
> ```
> 이렇게 배열을 `.map()` 으로 돌려서 `<li>` 를 동적으로 만듭니다.
> 지금 HTML 을 잘 이해해 두면, React 가 "HTML 을 자동으로 찍어내는 도구" 라는 감이 분명히 잡힙니다.

## 📊 테이블: `<table>`

"행과 열로 된 데이터" 를 표현할 때 씁니다. 엑셀 스프레드시트를 생각하면 돼요.
상품 목록, 가격표, 비교표, 통계 등이 대표적입니다.

> ⚠️ **중요**: 테이블은 **"표 형태의 데이터" 전용** 입니다.
> 옛날에는 페이지 레이아웃(2단 구성 같은 것)을 `<table>` 로 짜기도 했지만, **지금은 절대 금지** 입니다.
> 레이아웃용은 CSS 의 flex / grid 를 씁니다. (나중에 CSS 에서 배웁니다)

### 기본 구조

```html
<table>
  <thead>
    <tr>
      <th>이름</th>
      <th>카테고리</th>
      <th>가격</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>사과</td>
      <td>과일</td>
      <td>$1</td>
    </tr>
    <tr>
      <td>시금치</td>
      <td>채소</td>
      <td>$2</td>
    </tr>
  </tbody>
</table>
```

- `<table>` : 표 전체
- `<thead>` : 표의 **머리(header) 행들**
- `<tbody>` : 표의 **몸통(body) 행들**
- `<tr>` : table row. **한 행(row)**
- `<th>` : table header cell. **머리 칸** (자동으로 굵게 정렬됨)
- `<td>` : table data cell. **일반 칸** (내용 칸)

### 왜 `<th>` 와 `<td>` 를 구분하나?

- 스크린 리더가 각 데이터 칸을 읽을 때 **"어느 머리 칸 밑의 값인지"** 를 함께 읽어 줄 수 있습니다.
  예: "사과, 카테고리: 과일, 가격: $1" 처럼.
- CSS 로 머리 칸만 따로 스타일 주기 쉬움.

### 행/열 합치기: `colspan`, `rowspan`

```html
<tr>
  <th colspan="2">이름 & 카테고리</th>
  <th>가격</th>
</tr>
```

- `colspan="2"` → **가로로 2칸을 합침.**
- `rowspan="2"` → **세로로 2칸을 합침.**

> 🔗 Thinking in React 에서 `<ProductCategoryRow>` 가 이 `colSpan` 을 쓰고 있었죠.
> 이제 저게 "카테고리 머리 행을 테이블 폭 전체에 걸쳐 그리는 것" 임이 이해될 거예요.

### 확인해 볼 것

- `<ol>` 에 번호가 자동으로 매겨지는지
- `<ul>` 에 동그라미 불릿이 찍히는지
- `<th>` 가 굵게 나오고 `<td>` 는 평범한지
- 테이블에 기본으로 테두리가 **안 그려진다** 는 점(이건 CSS 로 따로 그려야 합니다)

## 📚 정리

| 태그 | 역할 |
|---|---|
| `<ul>` | 순서 없는 목록 |
| `<ol>` | 순서 있는 목록 |
| `<li>` | 목록 항목 |
| `<table>` | 표 전체 |
| `<thead>` / `<tbody>` | 표의 머리/몸통 영역 |
| `<tr>` | 표의 한 행 |
| `<th>` | 머리 칸 |
| `<td>` | 일반 칸 |
| `colspan` / `rowspan` | 칸 합치기 |
