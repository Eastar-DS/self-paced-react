# self-paced-react 학습 키워드 한줄 요약

각 키워드 옆에 한 줄 설명을 붙인 버전. 시험지처럼 훑기 좋아요.

---

## 1. TS 전환 중 터진 문제

- **`.tsx` 전환 시 `index.html` script 경로 문제** — `main.jsx` → `main.tsx`로 바꿨으면 `<script src>`도 같이 수정해야 함. VS Code Rename 기능으로 하면 자동 업데이트됨.

## 2. 모듈 선언

- **`declare module "*.png" { const src: string; export default src; }`** — TS에게 "PNG import하면 default export가 string(경로)"라고 알려주는 타입 선언. 없으면 TS2307 에러.
- **`vite/client` reference types** — 위 선언을 수동으로 안 쓰고 `/// <reference types="vite/client" />` 한 줄로 `.png`/`.svg`/`.css?raw` 등을 한꺼번에 해결하는 실무 관용구.

## 3. 유니온 타입 "이쁘게" 선언하기

- **`as const` (Const Assertion)** — 값을 가장 좁은 리터럴 타입으로 고정하고 readonly로 만듦. `["한식", "중식"]`의 타입을 `string[]`이 아니라 `readonly ["한식", "중식"]`으로 좁힘.
- **`typeof arr[number]` (Indexed Access Type)** — 배열 타입에서 "숫자 인덱스로 꺼낼 수 있는 모든 원소의 타입"을 추출. `number`는 숫자 리터럴이 아니라 "모든 인덱스"를 의미하는 타입.
- **SSOT (Single Source of Truth)** — "같은 정보는 한 곳에서만 정의하자"는 **설계 원칙**(TS 전용 개념이 아님. DB 정규화, `.env`, 디자인 토큰 등 넓은 맥락에 적용). 위의 `as const` + `typeof arr[number]` 조합은 **이 원칙을 TS에서 구현하는 한 가지 기법**이지, SSOT 자체는 아님.
- **`type` vs `interface` export 규칙** — 둘 다 `export` 키워드를 명시해야 다른 파일에서 import 가능. "알아서 export"되는 건 없음.

## 4. Props 타입 설계

- **setter 시그니처 (DOM 이벤트 vs 앱 콜백)** — `onChange`는 DOM 이벤트(`ChangeEvent`)를 받고, `onCategoryChange` 같은 앱 콜백은 도메인 값(`FilterCategory`)을 받음. 서로 다른 레이어.
- **마우스 호버로 타입 확인** — VS Code에서 변수나 핸들러 파라미터에 커서 올리면 추론된 타입이 툴팁으로 뜸. 타입을 "검색"하는 게 아니라 "IDE가 알려주는 걸 읽는" 습관.
- **Contextual Typing** — 인라인 화살표 함수(`onChange={(e) => ...}`)의 `e` 타입을 TS가 부모 prop 시그니처에서 자동 추론. 그래서 명시할 필요 없음.
- **구조분해 `:` vs 타입 annotation `:`** — `{ onClose }` 안의 `:`는 JS rename 문법, `}: Props`의 `:`는 TS 타입 annotation. `{ onClose: () => void }`처럼 쓰면 JS가 `onClose`를 `() => void`로 rename하려 함(에러).

## 5. 객체 타입 자동 추론

- **Object literal key inference** — `const obj = { 한식: x, 중식: y, ... }`의 타입이 자동으로 리터럴 키로 추론돼서 유니온으로 인덱싱해도 에러가 안 남. 명시적 `Record` 없이도 동작.
- **`Record<K, V>` 완전성 검사** — 타입 명시하면 유니온의 모든 키가 정의돼 있는지 "정의 시점"에 체크해줌. 없으면 "사용 시점"(인덱싱할 때)에 에러가 늦게 뜸.

## 6. useState 제네릭 & 캐스팅

- **제네릭 생략 vs 명시 기준** — 초기값이 최종 타입과 같으면 생략(`useState("")` → `string`), 다르면 명시(`useState<Restaurant[]>([])` — 초기값 `[]`이 `never[]`로 추론되는 걸 막기 위해).
- **상태 타입 범위 설계 (`RestaurantCategory | ""`)** — 초기값이 `""`고 나중에 도메인 값이 담길 거면 유니온으로 정직하게 두는 게 안전.
- **`as` 캐스팅의 의미** — "이 값의 타입은 이거야, 내가 책임질게"라고 TS에게 선언. 거짓말하면 런타임에 터짐.
- **`as typeof state` 패턴** — setter 호출 시 state 타입 그대로 재활용. 타입 바뀌면 한 곳만 고치면 됨.

## 7. Props 스타일

- **구조분해 vs `props` 객체** — 구조분해는 컴포넌트 API를 시그니처에 드러내고 `props.` 반복을 제거함. 기본값, rename, rest 모두 구조분해 전제. 다만 `{...props}` 스프레드 wrapper류에선 `props` 객체가 편함.
- **Rest Operator / `{...props}` 스프레드** — `const Input = ({ label, ...rest }) => <input {...rest} />` 형태. 특정 props는 뽑아 쓰고 나머지는 자식에게 통째로 넘김. 공통 Input/Button 만들 때 필수.

## 8. `React.` 네임스페이스 vs named import

- **React 17+ Automatic JSX Runtime** — `"jsx": "react-jsx"` 설정이면 파일마다 `import React` 쓸 필요 없음. 컴파일러가 자동으로 jsx-runtime을 import.
- **Named import가 모던 표준인 이유** — `import React`를 안 해도 되는 시대가 오면서 `React.X`로 쓸 이유가 사라짐. `import { ReactNode, ChangeEvent } from "react"`가 공식 문서와 신규 프로젝트 표준.

## 9. children과 합성 패턴

- **JSX → `React.createElement` 변환** — `<Modal>hello</Modal>` = `React.createElement(Modal, {}, "hello")`. 세 번째 이후 인자가 `children` prop에 자동 주입됨.
- **`children`은 "특별한 이름의 prop"** — 문법만 다르게 생겼을 뿐 다른 prop과 동일. `<Modal children="hi" />`로 써도 동작은 같음.
- **Composition (합성)** — 상속 없이 재사용하는 React 고유 방식. "안에 무엇이 들어갈지"를 호출하는 쪽이 결정하게 해서 유연성 확보.
- **공용 컴포넌트 사용 위치 (말단 위임)** — 공통 Modal은 최상위(App)가 아니라 말단 컴포넌트(AddRestaurantModal, RestaurantDetailModal) 내부에서 사용. App은 "도메인 모달"만 알고, 껍데기는 모름.

## 10. 재사용 Modal 패턴 비교

- **children prop 패턴** — `<Modal>{내용}</Modal>`. 가장 단순, 가장 범용. 레벨2 기본.
- **Slots 패턴** — `<Modal header={...} body={...} footer={...} />`. 구조 강제, prop 많아짐.
- **Compound Component** — `<Modal><Modal.Header/>...</Modal>`. Radix/Chakra/MUI가 쓰는 방식. Context 필요.
- **Render Props** — `<Modal>{({close}) => <Button onClick={close}/>}</Modal>`. 커스텀 훅이 나오면서 거의 퇴장.
- **Headless Library** — Radix UI, Headless UI 같은 라이브러리가 접근성/포커스 트랩/ESC를 해결해주고 스타일만 우리가 입힘. 실무 신규 프로젝트 기본값.
- **`role="dialog"` ≠ 재사용 패턴** — 재사용성은 "구조 설계" 문제, `role`은 "접근성(a11y)" 속성. 다른 축이라 둘 다 해야 함.

## 11. useEffect 이해 점검

- **Side Effect 정의** — 렌더링 바깥의 효과(DOM 조작, 타이머, 서버 통신, localStorage 등). React가 화면 그리는 일 외에 일어나는 모든 것.
- **콜백 전달 (함수 실행 ❌)** — `useEffect(fn())` ❌ / `useEffect(() => { ... })` ✅. 함수를 "즉시 실행한 결과"가 아니라 "나중에 실행할 함수 자체"를 넘김.
- **렌더링 이후 실행 시점** — 컴포넌트가 화면에 그려진 뒤 비동기로 실행. 렌더를 막지 않음.
- **Cleanup 실행 시점 (unmount + 재실행 직전)** — unmount 시 한 번뿐만 아니라, 의존성 바뀌어서 effect 재실행되기 직전에도 실행됨. "이전 설치물을 치우고 새로 설치" 구조.
- **의존성 배열 의미 ("참조 값 목록")** — "재실행 조건"이 아니라 "effect 내부에서 사용하는 외부 값 목록". 빠진 게 있으면 stale closure 버그.
- **빈 배열 `[]` 의미** — "이 effect는 외부 값을 하나도 참조하지 않음" = 마운트 시 한 번만 실행.
- **`react-hooks/exhaustive-deps` 린트 룰** — ESLint가 의존성 누락을 자동 탐지. 실무에선 이 경고를 무시하지 않는 게 기본 룰.

## 12. Stale Closure

- **React에서 Stale Closure 발생 시나리오** — `useEffect`의 의존성 배열에서 값을 누락하면, 콜백이 마운트 시점의 옛 props/state를 붙잡고 있어서 최신값이 반영 안 됨. 예: `[count]`를 빠뜨리면 카운터가 항상 0으로 찍힘.
- **의존성 배열로 해결 원리** — 의존성이 바뀌면 cleanup → effect 재실행이 일어나면서 콜백이 최신값을 붙잡은 새 버전으로 교체됨. 그래서 배열에 "effect가 참조하는 모든 값"을 빠짐없이 넣어야 함.
